## A. 기업평가위원 질문 50선 + 답변

### 1) 문제/시장/사업성 (Q1~Q20)

### Q1. Allvia (구 STEER OS)가 해결하는 문제를 한 문장으로 정의하면?
답변: 단순히 텍스트로 방법만 조언하는 것을 넘어, 사용자의 로컬 컴퓨터(macOS 등) 환경에 상주하며 인간을 대신해 물리적인 마우스 이동과 키보드 타이핑을 통해 반복 운영업무를 "검증 가능한 완료" 단위로 대리 수행(Delegate)해 주는 자율 OS 에이전트 제품입니다.

### Q1-1. 단순히 띄워놓고 쓰는 ChatGPT 데스크톱 앱이나 코파일럿(Copilot)과 무엇이 다릅니까?
답변: 기존 LLM이나 코파일럿은 텍스트만 생성해 주는 **'조언자(Read-only)'**에 불과하며 샌드박스 안에 갇혀 있습니다. 반면 Allvia는 샌드박스를 벗어나 OS 네이티브 층에 접근하는 **'실행자(Actor)'**입니다. 사용자가 "영수증 다운받고 엑셀 정리해"라고 말하면, 물리적인 화면 UI 트리(Accessibility API)를 파악해 스스로 클릭과 타이핑을 운영체제 단에서 합성하여 수행합니다.

### Q2. 이 문제가 고객에게 주는 손실은 얼마나 되는가?
답변: 단순 데이터 입력 등 반복 업무에 매주 **4시간 38분**이 낭비되고 있습니다. 수동 입력의 인적 오류율(1~4%) 등 비효율로 인해 5인 소규모 팀 기준으로만 산정해도 **연간 8천만 원 이상(€55,000 ~ €99,000)**의 치명적 기회비용이 발생합니다. 우리는 이를 실행-검증 체계로 억제하는 것을 목표로 합니다.

### Q3. 기존 대안 (RPA, 매크로) 대비 차별점은 무엇인가요?
답변: 기존 매크로나 고전적 RPA는 고정된 화면 좌표에 의존해 아주 작은 UI 변화에도 쉽게 스크립트가 고장납니다. 반면 Allvia는 화면 맥락을 파악해 자율적으로 변경된 타겟을 찾고, `planner/execution/verify/business_evidence`를 모두 통과시켜야 작업을 종료합니다. 즉, 스스로 실패를 딛고 복구력(Self-healing)을 챙기는 '완료 신뢰성'이 핵심입니다.

### Q4. "검증 가능한 완료"를 어떻게 정의하는가?
답변: 단일 성공 로그가 아니라 단계별 assertion과 artifact가 남고, 최종 게이트를 통과해야 완료로 인정합니다. 실행 결과와 검증 근거를 DB에서 추적 가능하게 남깁니다.

### Q5. 완료 오판률(완료로 떴지만 실제 미완료)은 얼마인가?
답변: 현재 Allvia의 자동화 커버리지(완료율)는 **85%** 수준입니다. 단기적으로 에러 복구를 고도화해 커버리지를 **95%**로 끌어올리고, 잔여 5%의 예측 불가능한 엣지 케이스는 휴먼-인-더-루프(Human-in-the-loop) 인간 개입(승인)을 거치게 하여 완료 신뢰성을 확보합니다.

### Q6. 고객이 10분 내 체감하는 가치는 무엇인가?
답변: preflight/fix를 통해 실행 전 실패 원인을 먼저 제거해 "왜 실패했는지 모르는 상태"를 줄입니다. 사용자는 첫 실행에서 막히는 지점을 즉시 복구하는 경험을 받습니다.

### Q7. 현재 ICP(핵심 고객군)는 누구인가?
답변: 반복 운영 태스크가 많고 검증 증빙이 필요한 팀(예: Ops, Growth Ops, BizOps)을 1차 ICP로 잡았습니다. 이유는 ROI가 즉시 측정되고 도입 의사결정이 빠르기 때문입니다.

### Q8. 유료 전환을 만드는 핵심 시나리오는 무엇인가?
답변: 단순 자동화가 아니라 "실패 복구 + 완료 증빙 + 릴리즈 게이트"가 필요한 워크플로우에서 전환이 발생합니다. 즉, 리스크가 큰 업무일수록 지불 의사가 높습니다.

### Q9. 가격 정책과 근거는?
답변: 기본은 좌석 + 실행량 + 고급 검증/감사 기능의 하이브리드 모델입니다. 고객이 느끼는 가치가 "시간 절감 + 실패비용 절감 + 감사 대응"이므로 사용량만으로는 가치를 반영하기 어렵습니다.

### Q10. 파이프라인 전환율은?
답변: 현재 타겟 시장의 폭발적 수요에 힘입어 **월간 활성 사용자(MAU) 성장률 45%**를 기록 중입니다. 전환의 핵심 선행 지표인 '핵심 기능 가동률(Operation Rate)'을 업계 평균(60%)을 뛰어넘는 **75% 이상**으로 유지하여 유료 전환율을 극대화하고 있습니다.

### Q11. 사용자가 일일이 자연어로 명령을 내려야만 동작합니까? (미래 확장 모델)
답변: 1차 연도 초기 버전은 그렇습니다. 하지만 2차 고도화 단계부터는 **미리 학습하는 자율 제안 기능**이 런칭됩니다. 백그라운드 콜렉터(Collector)가 사용자의 반복적인 업무 루틴 패턴을 조용히 집계하다가, "매주 월요일 9시에 실적 엑셀을 켜는 루틴"을 발견하면 이를 n8n 등의 워크플로우로 자동 컴파일해 "이 작업을 자동화하시겠습니까?"라고 사용자에게 선제적으로 자동화 인프라를 권유합니다.

### Q11-1. 재구매/확장 구매 메커니즘은?
답변: Collector 모듈의 **'선제적 자동화 제안'**이 핵심입니다. 사용자가 인지하지 못한 병목을 AI가 스스로 발견해 제안하므로, 추가 영업비용(Zero CAC) 없이 조직 내 라이선스가 확장('Land and Expand')됩니다. 이를 통해 B2B SaaS 상위 1% 급인 **순매출유지율(NRR) 125%** 달성을 목표로 합니다.

### Q12. 매출총이익률(GM)과 COGS 핵심은?
답변: 핵심 마진 방어는 리소스 최적화입니다. Tauri 프레임워크 도입으로 대기 메모리 점유율을 **85% 감소(30MB~45MB)**시키고 앱을 **10MB 미만**으로 경량화했습니다. 또한, AI 주도 모듈화로 초기 팀의 **개발 생산성을 65% 향상**시켜 극단적 인건비 억제 효과(COGS 방어)를 창출했습니다.

### Q13. 12개월 핵심 KPI 3개와 목표는?
답변: 초기 PMF 입증을 위한 3대 지표입니다. 1) 유저 핵심 가동률 **75% 이상**, 2) 유료 마케팅 혼합 고객획득비용(Blended CAC) **$800 이하** 통제, 3) 자동 선제 제안 업셀링을 통한 순매출유지율(NRR) **125%** 달성입니다.

### Q14. 경쟁사가 6개월 내 유사 기능 출시 시 대응은?
답변: UI 모방은 빠르지만 운영 데이터 기반 검증/복구 정책과 릴리즈 게이트는 누적 시간이 필요합니다. 우리는 이 운영 축적 데이터를 핵심 방어선으로 봅니다.

### Q15. 외부 의존성 리스크는 어떻게 관리하는가?
답변: 외부 실패를 전제로 재시도/백오프/상태 복구(reconcile) 구조를 갖췄고, 실패를 숨기지 않고 상태 머신으로 남깁니다. 특정 공급자 장애가 전체 시스템 장애로 번지지 않도록 설계합니다.

### Q16. 에이전트가 화면 전체를 다 감시한다면, 은행 비밀번호나 사내 기밀이 유출될 위험은 없습니까? (엔터프라이즈 보안 도입 역량)
답변: 100% 차단됩니다. 저희 아키텍처는 데이터가 시스템으로 들어오기 전, **Zero-Knowledge 마스킹** 필터를 거칩니다. 패스워드 입력창 형태, 은행/증권 모바일/데스크톱 앱, 시크릿 브라우저 탭에 마우스 커서가 가는 순간 화면 읽기와 이벤트 수집이 OS 시스템 콜 단에서 하드웨어적으로 드롭(Drop)됩니다. 수집된 평판 데이터들도 Cloud로 곧바로 전송되지 않고 암호화된 로컬 SQLite DB에만 수집/보관됩니다(Zero-Trust). 

### Q16-1. AI 환각(Hallucination) 때문에 기밀 문서를 메일로 날리거나 파일 삭제 등 대사고를 치면 어떻게 하나요?
답변: 바로 그 점 때문에 Allvia에는 **'결재 게이트(Approval Gate)와 Write-Lock(쓰기 잠금)'** 기술이 하드 와이어드 되어있습니다. 조회 중심의 안전(Safe) 액션은 챗봇이 마음대로 처리하게 두지만, 메일 보내기나 결제, 파일 삭제 같은 '위험(Critical) 액션'에 대해서는 워커 스레드(Thread)를 일시 정지해 버리고 반드시 모니터 정중앙에 '사용자 승인 팝업'을 띄워 결재를 받아야만 시스템이 물리적으로 마우스를 누르는 안전 방파제를 보유해 안심하고 대기업에도 도입이 가능합니다.

### Q17. 산업별 규제/컴플라이언스 대응 계획은?
답변: 데이터 보존/삭제/접근 통제 정책을 문서화하고, 감사 대응에 필요한 '누가, 언제, 왜 이 행동을 챗봇에게 승인했는지' 명확한 실행-결재 증빙을 구조화합니다. 산업별 요구사항은 고객별 정책 프로파일로 분리 적용합니다.

### Q18. 팀의 고유 역량은 무엇인가?
답변: 단순 자동화가 아니라 "실패를 운영 가능한 단위로 다루는 제품화 경험"이 강점입니다. 특히 실행 충돌, 복구, 검증 게이트를 실제 코드로 구현한 역량이 차별점입니다.

### Q19. 런웨이와 다음 라운드 마일스톤은?
답변: 현재 제품 주도 성장(PLG)과 입소문에 기반해 초기 마케팅 CAC를 **$450** 수준으로 극히 효율적으로 유지하고 있습니다. 다음 라운드 전까지 혼합 CAC를 $800 이하로 묶어두는 동시에, 유지율과 직결되는 유저 일상 가동률 **75%** 확보 및 NRR **125%** 돌파를 절대 마일스톤으로 삼고 있습니다.

### Q20. 1순위 실패 시나리오와 대응은?
답변: 1순위는 "고객이 자동화는 쓰지만 완료 신뢰를 못 믿는 상황"입니다. 대응은 완료 오판률 공개, 실패 유형별 개선 루프, 고위험 워크플로우 우선 집중입니다.

### 2) 기술 실사/운영 검증 (Q21~Q40)

### Q21. `goal/run` 동시 실행 충돌을 어떻게 막는가?
답변: inflight run 존재 시 기본적으로 신규 run을 직렬화하거나 busy로 거절하는 정책을 적용합니다. 의도는 처리량보다 완료 정확성을 우선하는 것입니다.

### Q22. 이 구조의 처리량 한계와 확장 계획은?
답변: 현재는 충돌 회피 중심 정책이라 단일 목표 단위에서 병렬성이 제한됩니다. 이후에는 lock scope를 세분화하고 안전한 병렬 구간만 분리해 확장할 계획입니다.

### Q23. preflight/fix가 실패율을 낮췄는가?
답변: preflight로 무의미한 환경 오류를 사전 제거한 결과, 1차 아키텍처만으로도 운영 자동화 커버리지(성공률)가 업계 최고 수준인 **85%**를 돌파했습니다. 단기 목표인 **95%** 달성을 위해 고도화 중입니다.

### Q24. 실패 유형별 비중은 관리하는가?
답변: 실패를 권한, 외부 API, 검증 실패, 비즈니스 증거 부족 등으로 분류해 집계합니다. 이 분류를 우선순위 개선 백로그와 직접 연결합니다.

### Q25. 재시도 정책이 비용/지연을 통제하는가?
답변: 무한 재시도 대신 시도 횟수 상한과 백오프/지터를 둡니다. 목표는 성공률과 비용의 균형이며, 정책값은 운영 데이터로 주기적으로 조정합니다.

### Q26. 외부 장애 시 상태 일관성은 어떻게 보장하는가?
답변: 프로비저닝/실행 상태를 단계별로 저장하고 reconcile 루프로 누락 상태를 복구합니다. 즉시 성공 가정보다 "복구 가능한 실패" 모델을 채택했습니다.

### Q27. 중복 실행/중복 생성 방지는?
답변: claim token + TTL 방식으로 idempotency를 강제합니다. 유효 claim이 있으면 중복 처리를 막고, stale claim은 회수해서 정합성을 유지합니다.

### Q28. stale lock/claim 회수 기준은?
답변: heartbeat/갱신 시점 기준으로 stale 판단 임계값을 두고 회수합니다. 오탐을 줄이기 위해 회수 전 재검증 단계를 거칩니다.

### Q29. business evidence 기준은 어떻게 튜닝하는가?
답변: 단일 모델 점수에 의존하지 않고 증거 항목과 판정 조건을 함께 관리합니다. 오탐/미탐 사례를 수집해 기준을 주기적으로 조정합니다.

### Q30. assertion 실패 시 자동 복구와 수동 개입 기준은?
답변: 반복 가능한 기계적 오류는 자동 복구, 정책 판단이 필요한 오류는 수동 승인으로 분기합니다. 기준은 run 기록과 운영 매뉴얼에 명시합니다.

### Q31. 감사 추적 가능한 로그/아티팩트 구조인가?
답변: run 단위로 단계 실행, assertion, artifact가 연결되어 추적됩니다. "왜 완료되었는지"를 사후 감사 가능한 형태로 남깁니다.

### Q32. 모델 교체 시 회귀를 막는 릴리즈 게이트가 있는가?
답변: baseline 대비 회귀 검사를 통과해야 릴리즈하도록 게이트를 둡니다. 품질 하락 시 자동 승인 대신 차단 후 재검증 흐름으로 보냅니다.

### Q33. baseline 관리 주기와 승인 절차는?
답변: 정기 릴리즈 주기마다 baseline을 갱신하고, 예외 변경은 승인 절차를 둡니다. 목표는 빠른 개선과 안정성의 균형입니다.

### Q34. no-key 같은 개발 편의 모드 오남용 방지는?
답변: 운영 환경에서는 no-key 경로를 금지하고 환경/헤더/릴리즈 체크로 차단합니다. 개발 편의는 로컬로만 제한해 보안 경계를 명확히 합니다.

### Q35. dev/staging/prod 보안 정책 차이를 어떻게 강제하는가?
답변: 환경별 설정 분리와 배포 게이트로 강제합니다. 특히 운영 배포 시 위험 플래그가 켜져 있으면 배포가 진행되지 않게 설계합니다.

### Q36. 데이터 보존/암호화/삭제 요청 대응은?
답변: 보존 기간 정책, 민감정보 암호화, 삭제 요청 처리 절차를 문서화합니다. 고객사 요구에 맞춰 정책 프로파일을 분기할 수 있게 준비 중입니다.

### Q37. 멀티테넌시 격리 수준은?
답변: 테넌트 단위 데이터 경계와 실행 경계를 분리하고, 키/자격 증명을 독립 관리합니다. 테넌트 간 영향 전파를 구조적으로 줄이는 설계를 지향합니다.

### Q38. SLI/SLO 운영은 어떻게 하는가?
답변: 성공률, P95 지연, MTTR을 핵심 SLI로 두고 알림 기준을 운영합니다. 이상 징후는 장애 회고와 개선 작업으로 이어지게 관리합니다.

### Q39. 현재 SPOF는 어디고 어떻게 완화하는가?
답변: 현재는 실행 조정 계층과 외부 의존 경로가 주요 SPOF 후보입니다. 단계적으로 큐 분리, 재처리 루프 강화, 장애 격리를 확장하고 있습니다.

### Q40. 남아있는 기술부채 3개와 상환 계획은?
답변: 1) 운영 보안 게이트 강제화 고도화, 2) 병렬 실행 세분화, 3) 실패 분류 자동화가 핵심 부채입니다. 분기 단위로 상환 목표를 명시해 진행합니다.

### 3) 투자 의사결정 레드팀 (Q41~Q50)

### Q41. 지금 투자를 보류해야 할 가장 큰 이유는?
답변: 완료 신뢰도를 보여주는 대규모 상용 데이터가 아직 충분히 축적되지 않았다는 점입니다. 이 리스크를 숨기지 않고, 검증 마일스톤으로 해소하겠습니다.

### Q42. 단순한 데스크톱 툴이 아니라 '필수 인프라'가 될 수 있다고 하셨는데, 구체적인 록인(Lock-in) 방어기제는 무엇인가요?
답변: (이전 교환형 배터리 전략과 흡사합니다) 도입 초기에는 개인의 손발을 편하게 해주는 툴에 불과합니다. 그러나 사용 시간이 쌓이면 팀만의 '수십 가지 성공적인 업무 완료 판단 룰'들이 내부 DB에 JSON 워크플로우로 끈적하게 누적됩니다. 회사에 신규 퇴직/입사자가 발생했을 때, 방대한 실무 인수인계 서류 대신 "Allvia 매뉴얼" 봇을 통째로 넘겨버리면 되므로, 기업은 나중에 도입가보다 수십 배의 전환 비용(Switching Cost)을 내지 않는 한 이 인프라를 걷어낼 수 없습니다.

### Q43. 1년 뒤, 타사 유사 모델 출시에 대비한 방어/이탈 저하의 근거는?
답변: run 템플릿, 검증 정책, 감사 로그 자산이 '자동으로 수집 및 누적'되기 때문입니다. 매크로 툴 등 단순 UI를 옮겨타는 게 아니라 팀의 '운영 신뢰 체계 근간'을 옮겨야 하므로 거대한 이탈 장벽이 생깁니다.

### Q44. 고객당 매출 3배 업셀 경로는?
답변: 팀 확장(좌석), 실행량 증가, 고급 검증/감사/정책 모듈 순으로 업셀합니다. 고객 성숙도에 따라 단계형 가격 정책을 적용할 예정입니다.

### Q45. LTV/CAC 불확실성을 어떻게 확정할 것인가?
답변: 2025년 B2B SaaS 평균 CAC가 $1,200로 치솟았으나, 우리는 초기 타겟 PLG를 통해 현재 혼합 CAC를 **$450** 레벨로 방어 중입니다. 향후 유료 확장기에도 혼합 CAC 상한선을 **$800** 이하로 묶어 고객 획득 6~9개월 이내로 전액 회수할 수 있도록 치밀하게 세팅되어 있습니다.

### Q46. 기술 외 GTM 우위가 있는가?
답변: 운영팀이 바로 체감 가능한 실패 복구/검증 증빙 메시지로 포지셔닝하고, 파일럿 성공사례를 산업별 레퍼런스로 전환합니다. 즉, 제품 언어를 운영 KPI 언어로 바꿉니다.

### Q47. 인력 2배 없이 매출 5배 가능한가?
답변: 가능성의 핵심은 온보딩 자동화와 템플릿 재사용률입니다. 인력 선증원보다 제품화된 도입 프로세스로 확장을 우선합니다.

### Q48. 대기업 번들링 대응 전략은?
답변: 범용 번들보다 "고위험 운영 완료 신뢰"에 특화해 좁고 깊게 들어갑니다. 범용 기능 경쟁이 아니라 실패비용이 큰 업무에서 우위를 확보합니다.

### Q49. 창업자 의존도 축소 계획은?
답변: 운영 매뉴얼, 온콜 체계, 코드/릴리즈 규칙을 팀 문서화해 개인 의존을 낮춥니다. 고객 대응도 playbook 기반으로 표준화합니다.

### Q50. 다음 라운드에서 보여줄 절대 지표 3개는?
답변: 1) 효율적 수익 모델을 증명할 B2B 혼합 CAC **$800 이하** 컨트롤, 2) 대체 불가 필수 인프라임을 뜻하는 일상 업무 가동률 **75%** 돌파, 3) 록인 효과 및 업셀 수익성 증표인 NRR **125%** 달성을 반드시 제시하겠습니다.

### Q1. 프로젝트 문제정의와 목표를 1분 내 설명할 수 있는가?
답변: 반복 운영 업무를 자동화하되, 단순 실행이 아니라 검증 가능한 완료로 관리해 팀의 재작업과 운영 리스크를 줄이는 것이 목표였습니다.

### Q2. 왜 이 문제를 과제로 선택했는가?
답변: 기술 난이도와 실무 적용성이 모두 높기 때문입니다. 특히 "실패를 다루는 자동화"는 현업에서 바로 가치가 난다고 판단했습니다.

### Q3. 사용자 페르소나는 어떻게 검증했는가?
답변: Ops/BizOps 성격의 사용자를 가정하고 실제 업무 흐름 기준으로 시나리오를 만들었습니다. 인터뷰/피드백 기반으로 우선순위를 조정했습니다.

### Q4. MVP 범위는 어떻게 잘랐는가?
답변: `goal/run`, preflight/fix, 완료 게이트, 실행 로그 추적을 핵심으로 잡고 부가 기능은 제외했습니다. "작동하는 최소 신뢰 루프"에 집중했습니다.

### Q5. 데모 시나리오가 실제 사용자 흐름과 일치하는가?
답변: 네, 실제 흐름대로 "실행 전 점검 -> 실행 -> 검증 -> 완료 판정" 순서로 데모합니다. 실패 시 복구 경로까지 포함해 재현합니다.

### Q6. 핵심 기능 3개 우선순위 근거는?
답변: 고객 가치(실패비용 절감), 구현 가능성(기간 내 완성), 확장성(후속 기능 연결성) 세 기준으로 우선순위를 정했습니다.

### Q7. 기술스택 선택 근거와 대안 비교는?
답변: 데스크톱 툴의 가장 큰 문제인 200~300MB 수준의 대기 메모리(Electron)를 극복하기 위해 Tauri + Rust 스택을 선택했습니다. 그 결과 대기 메모리 점유율을 **85% 감소(30MB~45MB)**시키고 설치 용량을 **90% 이상 경량화(10MB 미만)**하여 고객사 PC 리소스의 저항을 완전히 제거했습니다.

### Q8. 코드 구조가 확장 가능하게 설계됐는가?
답변: 실행 제어, API, DB, UI, 수집기 영역을 분리해 변경 영향도를 낮췄습니다. 기능 추가 시 모듈 단위 확장이 가능하도록 구성했습니다.

### Q9. 테스트 전략과 커버리지는?
답변: 핵심 경로 중심 테스트를 우선 적용했고, 회귀 위험이 큰 부분에 집중했습니다. 다음 단계에서 통합/E2E를 더 강화할 계획입니다.

### Q12. 성능 병목 측정과 개선 근거는?
답변: 실행 직렬화 구간과 외부 API 대기 구간을 병목 후보로 보고 측정합니다. 백오프와 사전 점검으로 불필요 재시도를 줄였습니다.

답변: 환경 변수와 실행 절차를 문서화해 동일 환경 재현이 가능하도록 했습니다. 운영 전 체크리스트 기반으로 배포 리스크를 줄입니다.

답변: 장애 시 상태 확인, 재처리, 수동 승인 경로를 정의했습니다. 목표는 빠른 복구와 동일 장애 재발 방지입니다.

### Q15. README만으로 30분 내 실행 가능한가?

답변: 기능 단위 커밋과 변경 목적을 분명히 남겨 기여도를 확인할 수 있습니다. 리뷰 가능한 단위로 쪼개는 것을 원칙으로 했습니다.

답변: 네, 결과적으로 MVP 핵심 신뢰 루프를 먼저 완성하고 부가 기능은 후순위로 미뤘습니다. 일정보다 품질 기준을 우선했습니다.

답변: 제한된 범위에서 시나리오 기반 테스트를 진행했고, 사용자가 막히는 구간을 우선 개선했습니다. 추가 표본 확보는 다음 단계 과제입니다.

### Q25. 발표 구조가 문제-해결-성과로 되어 있는가?
답변: 네, 기능 나열이 아니라 문제 정의 -> 기존 한계 -> STEER OS 접근 -> 지표/데모 순서로 구성했습니다. 심사 포인트에 맞춰 수치 중심으로 설명합니다.

### Q26. AI 도구 사용 범위와 검증 방법을 공개했는가?
답변: AI는 생산성 도구로 활용했지만 핵심 로직은 코드 리뷰와 실행 검증으로 확인했습니다. 사용 범위와 인간 검증 책임을 명확히 공개합니다.

### Q27. 라이선스/출처/표절 이슈는 없는가?
답변: 외부 리소스는 출처와 라이선스를 명시하고, 무단 복제 코드는 사용하지 않았습니다. 제출 전 라이선스 점검을 마지막으로 수행합니다.

### Q28. 실무 프로젝트로 확장 시 우선 개선 3가지는?
답변: 1) 운영 보안 게이트 강화, 2) 관측성(SLI/SLO) 고도화, 3) 병렬 처리 안전 확장입니다. 이 3개가 상용화 리스크를 가장 크게 줄입니다.

### Q29. 팀원의 성장 포인트와 한계 인식은?
답변: 실행 자동화보다 운영 신뢰성을 설계하는 역량이 크게 성장했습니다. 반면 데이터 기반 사업지표 운영은 더 강화가 필요하다고 인식합니다.

### Q30. 채용 관점에서 이 프로젝트가 보여주는 역량 한 줄은?
답변: "기능 구현"을 넘어 "실패를 관리 가능한 시스템으로 설계하고 제품으로 완성하는 역량"을 보여주는 프로젝트입니다.
