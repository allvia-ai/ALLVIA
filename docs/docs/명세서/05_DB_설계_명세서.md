## 1. 데이터베이스 개요

### 1.1 DB 엔진 및 운영 설정

| 항목 | 값 | 비고 |
|------|------|------|
| 엔진 | SQLite 3 | 로컬 임베디드 (rusqlite bundled) |
| 모드 | WAL (Write-Ahead Logging) | 읽기/쓰기 동시성 향상 |
| 기본 경로 | `./steer.db` | `STEER_DB_PATH`로 오버라이드 가능 |
| Busy Timeout | 5,000ms | 동시 접근 대기 |
| 배치 Insert | 100건 / 1,000ms flush | 쓰기 성능 최적화 |
| Insert 재시도 | 3회 / 50ms backoff | 실패 복구 |
| DB 크기 상한 | 500MB | 초과 시 오래된 데이터 삭제 (`retention.max_db_mb`) |
| VACUUM 주기 | 24시간 | 디스크 공간 회수 (`retention.vacuum_hours`) |
| Validation Level | lenient | 검증 수준 (`validation_level`) |

### 1.2 SQLite 선택 근거

| 기준 | SQLite | PostgreSQL/MySQL | 선택 이유 |
|------|--------|-----------------|-----------|
| 설치 복잡도 | 제로 (임베디드) | 서버 설치 필요 | 개인 데스크톱 앱에 서버 DB 불필요 |
| 배포 크기 | ~1MB (bundled) | ~50MB+ | 경량 데스크톱 앱 요구 |
| 동시 사용자 | 1명 (로컬) | 다수 | 단일 사용자 로컬 에이전트 |
| 성능 | 읽기 극도로 빠름 | 쓰기 병행성 우수 | 읽기 위주 + WAL로 쓰기 충분 |
| 백업 | 파일 복사 | 덤프 명령 필요 | 단순 파일 복사로 완전 백업 |

### 2.1 events — 원시 이벤트 저장

**파일**: `migrations/001_init.sql`
**용도**: OS 센서에서 수집된 모든 원시 이벤트를 저장

#### 실제 DDL
```sql
CREATE TABLE IF NOT EXISTS events (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    schema_version TEXT NOT NULL,
    event_id TEXT NOT NULL,
    ts TEXT NOT NULL,
    source TEXT NOT NULL,
    app TEXT NOT NULL,
    event_type TEXT NOT NULL,
    priority TEXT NOT NULL,
    resource_type TEXT NOT NULL,
    resource_id TEXT NOT NULL,
    payload_json TEXT NOT NULL,
    privacy_json TEXT NOT NULL,
    pid INTEGER,
    window_id TEXT,
    raw_json TEXT NOT NULL,
    ingested_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_events_ts ON events (ts);
CREATE INDEX IF NOT EXISTS idx_events_app ON events (app);
CREATE INDEX IF NOT EXISTS idx_events_event_type ON events (event_type);
```

#### 컬럼 상세

| 컬럼 | 타입 | 제약 | 설명 |
|------|------|------|------|
| `id` | INTEGER | PK AUTOINCREMENT | 자동 증가 ID |
| `schema_version` | TEXT | NOT NULL | 이벤트 스키마 버전 ("1.0") |
| `event_id` | TEXT | NOT NULL | 고유 이벤트 UUID |
| `ts` | TEXT | NOT NULL | 이벤트 발생 타임스탬프 (ISO 8601) |
| `source` | TEXT | NOT NULL | 이벤트 소스 (`os`, `addin`) |
| `app` | TEXT | NOT NULL | 애플리케이션명 (`chrome.exe` 등) |
| `event_type` | TEXT | NOT NULL | 이벤트 유형 (`os.foreground_changed` 등) |
| `priority` | TEXT | NOT NULL | 우선순위 (`P0`, `P1`, `P2`) |
| `resource_type` | TEXT | NOT NULL | 리소스 유형 (`window`, `file` 등) |
| `resource_id` | TEXT | NOT NULL | 리소스 식별자 |
| `payload_json` | TEXT | NOT NULL | 이벤트 페이로드 (JSON) |
| `privacy_json` | TEXT | NOT NULL | 프라이버시 처리 메타 (JSON) |
| `pid` | INTEGER | — (nullable) | 프로세스 ID |
| `window_id` | TEXT | — (nullable) | 윈도우 식별자 |
| `raw_json` | TEXT | NOT NULL | 원시 이벤트 전체 데이터 (JSON) |
| `ingested_at` | TEXT | NOT NULL DEFAULT now | 수집 시각 (자동 설정) |

#### 인덱스

| 인덱스명 | 대상 컬럼 | 용도 | 사용 쿼리 |
|----------|-----------|------|-----------|
| `idx_events_ts` | `ts` | 시간 범위 조회 | 세션 빌드, 리텐션 삭제 |
| `idx_events_app` | `app` | 앱별 필터링 | 대시보드 앱 통계 |
| `idx_events_event_type` | `event_type` | 이벤트 유형별 필터링 | 패턴 분석 |

#### 예시 데이터
```json
{
  "id": 1,
  "schema_version": "1.0",
  "event_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "ts": "2026-02-18T09:30:15",
  "source": "os",
  "app": "chrome.exe",
  "event_type": "os.foreground_changed",
  "priority": "P0",
  "resource_type": "window",
  "resource_id": "window_12345",
  "payload_json": "{\"title_hash\": \"abc123\", \"url_domain\": \"google.com\"}",
  "privacy_json": "{\"masked\": true, \"rules_applied\": [\"url_domain_only\"]}",
  "pid": 4567,
  "window_id": "win_001",
  "raw_json": "{...}",
  "ingested_at": "2026-02-18T09:30:15"
}
```

### 2.4 handoff_queue — 핸드오프 큐

**파일**: `migrations/004_handoff_queue.sql`
**용도**: 자동화 대상으로 패키징된 워크플로우 요청 큐

#### 실제 DDL
```sql
CREATE TABLE IF NOT EXISTS handoff_queue (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    package_id TEXT NOT NULL UNIQUE,
    created_at TEXT NOT NULL,
    status TEXT NOT NULL,
    payload_json TEXT NOT NULL,
    payload_size INTEGER NOT NULL,
    expires_at TEXT,
    error TEXT
);

CREATE INDEX IF NOT EXISTS idx_handoff_status ON handoff_queue (status);
CREATE INDEX IF NOT EXISTS idx_handoff_created_at ON handoff_queue (created_at);
```

#### 컬럼 상세

| 컬럼 | 타입 | 제약 | 설명 |
|------|------|------|------|
| `id` | INTEGER | PK AUTOINCREMENT | 자동 증가 |
| `package_id` | TEXT | NOT NULL UNIQUE | 패키지 고유 UUID |
| `created_at` | TEXT | NOT NULL | 생성 시각 |
| `status` | TEXT | NOT NULL | 상태 |
| `payload_json` | TEXT | NOT NULL | 워크플로우 페이로드 (JSON) |
| `payload_size` | INTEGER | NOT NULL | 페이로드 크기 (bytes) |
| `expires_at` | TEXT | — (nullable) | 만료 시각 |
| `error` | TEXT | — (nullable) | 에러 메시지 |

#### 상태 전이도
```
┌──────────┐   n8n 전송 성공   ┌──────────┐
│ pending  │────────────────→│   sent   │
└────┬─────┘                 └──────────┘
     │
     ├── 만료 시각 도달 ──→ expired
     │
     └── 전송 실패 ───────→ failed
```

| status | 설명 |
|--------|------|
| `pending` | 처리 대기 중 |
| `sent` | n8n에 전송 완료 |
| `expired` | 만료됨 |
| `failed` | 처리 실패 |

### 3.1 minute_aggregates — 분 단위 집약

| 항목 | 값 |
|------|------|
| 생성 | Rust Collector 시작 시 |
| 갱신 주기 | 5분 (`STEER_COLLECTOR_AGG_INTERVAL_SEC`, 기본 300초) |
| 용도 | 이벤트를 5분 단위로 집약하여 대시보드 빠른 조회 제공 |

```sql
-- 실제 스키마 (collector_rs.rs 기반)
CREATE TABLE IF NOT EXISTS minute_aggregates (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp TEXT NOT NULL,            -- 분 단위 타임스탬프
    app TEXT NOT NULL,                  -- 애플리케이션명
    event_count INTEGER DEFAULT 0,      -- 이벤트 수
    actions_json TEXT,                  -- 액션 상세 (JSON)
    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(timestamp, app)
);

CREATE INDEX IF NOT EXISTS idx_ma_ts ON minute_aggregates(timestamp);
```

### 3.2 daily_summaries — 일일 요약

| 항목 | 값 |
|------|------|
| 생성 | Rust Collector 배치 작업 |

### 5.2 쿼리 상세

#### Q1: 최근 N시간 이벤트 조회 (세션 빌드용)
```sql
SELECT * FROM events
WHERE ts >= datetime('now', '-6 hours')
ORDER BY ts ASC;
-- 인덱스: idx_events_ts → 범위 스캔
-- 예상 결과: ~1,250건 (6시간 × ~208건/시간)
```

#### Q2: 앱별 이벤트 집계 (대시보드)
```sql
SELECT app, COUNT(*) as cnt
FROM events
WHERE ts >= datetime('now', '-24 hours')
GROUP BY app
ORDER BY cnt DESC
LIMIT 10;
-- 인덱스: idx_events_ts (WHERE) + idx_events_app (GROUP BY)
```

#### Q3: 배치 Insert (100건 단위)
```sql
INSERT INTO events (schema_version, event_id, ts, source, app, event_type,
                    priority, resource_type, resource_id, payload_json,
                    privacy_json, pid, window_id, raw_json)
VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?);
-- 100건 트랜잭션 단위로 실행
-- WAL 모드에서 읽기 차단 없음
```

#### Q7: 리텐션 삭제 (배치)
```sql
DELETE FROM events
WHERE ts < datetime('now', '-7 days')
LIMIT 5000;
-- 배치 삭제로 락 시간 최소화
-- idx_events_ts 인덱스 활용
```

#### Q8: 상태 커서 읽기/갱신 (UPSERT)
```sql
-- 읽기
SELECT value FROM state WHERE key = 'last_session_cursor';

-- 갱신 (UPSERT)
INSERT INTO state (key, value, updated_at) VALUES (?, ?, datetime('now'))
ON CONFLICT(key) DO UPDATE SET value = excluded.value, updated_at = excluded.updated_at;
```

## 8. 백업 및 복구

### 8.1 백업 전략

| 방법 | 명령 | 주기 | 설명 |
|------|------|------|------|
| **파일 복사** | `cp steer.db steer.db.bak` | 일 1회 (권장) | WAL 모드에서도 안전한 복사 |
| **SQLite 백업 API** | `.backup steer_backup.db` | 필요 시 | 온라인 백업 (핫 백업) |
| **WAL 체크포인트 후 복사** | `PRAGMA wal_checkpoint(TRUNCATE)` → `cp` | 안전한 풀 백업 시 | WAL 파일 통합 후 복사 |

### 8.2 백업 스크립트 예시
```bash
#!/bin/bash
# steer_backup.sh — 일일 백업
BACKUP_DIR="$HOME/.steer/backups"
DB_PATH="./steer.db"
DATE=$(date +%Y%m%d_%H%M%S)

mkdir -p "$BACKUP_DIR"

# WAL 체크포인트 후 복사
sqlite3 "$DB_PATH" "PRAGMA wal_checkpoint(TRUNCATE);"
cp "$DB_PATH" "$BACKUP_DIR/steer_${DATE}.db"

# 7일 이상된 백업 삭제
find "$BACKUP_DIR" -name "steer_*.db" -mtime +7 -delete

echo "Backup completed: steer_${DATE}.db"
```

### 8.3 복구 절차

```
1. 현재 DB 상태 확인
   sqlite3 steer.db "PRAGMA integrity_check;"

2-A. 정상 → 백업 불필요

2-B. 손상 감지 시:
   a) 서비스 중지: kill.sh
   b) 백업 복원: cp ~/.steer/backups/steer_최신.db ./steer.db
   c) 무결성 검증: sqlite3 steer.db "PRAGMA integrity_check;"
   d) 서비스 재시작

2-C. 백업도 없는 경우:
   a) 서비스 중지
   b) DB 재초기화: PYTHONPATH=src python scripts/init_db.py --config configs/config.yaml
   c) 서비스 재시작 (데이터는 유실되지만 수집 재개)
```
