## 2. F01: Shadow — 백그라운드 데이터 수집

### 2.1 기능 개요
운영체제의 이벤트(앱 전환, 파일 변경, 유휴 상태)를 백그라운드에서 자동 감지하여 수집하는 핵심 데이터 수집 기능이다.

### 2.2 센서 구성

| 센서 | 파일 | 감지 대상 | 이벤트 타입 |
|------|------|-----------|-------------|
| Foreground | `sensors/os/windows_foreground.py` | 전면 앱 변경 | `os.foreground_changed` |
| Idle | `sensors/os/windows_idle.py` | 유휴 상태 진입/복귀 | `os.idle_start` / `os.idle_end` |
| File Watcher | `sensors/os/file_watcher.py` | 파일 생성/수정/삭제 | `os.file_created` / `os.file_modified` |
| Focus Blocker | `sensors/os/focus_blocker.py` | 앱 포커스 블로킹 이벤트 | `os.focus_blocked` |
| HTTP Receiver | `sensors/addins/receiver_http.py` | 외부 애드인 이벤트 (HTTP) | `addin.*` |
| Pipe Receiver | `sensors/addins/receiver_pipe.py` | 외부 애드인 이벤트 (Pipe) | `addin.*` |

### 2.3 데이터 처리 파이프라인

```
센서 이벤트 발생
    │
    ▼
Event Bus (bus.py) — 이벤트 큐 (max 1,000건)
    │
    ▼
우선순위 분류 (priority.py)
  ├─ P0: 최우선 (앱 전환 등)
  ├─ P1: 일반
  └─ P2: 배경 (큐 80% 이상이면 드롭)
    │
    ▼
정규화 (normalize.py) — 이벤트 포맷 통일
    │
    ▼
개인정보 필터 (privacy.py)
  ├─ Allowlist 앱만 통과 (27개 앱)
  ├─ Denylist 앱은 drop (keepass, 1password 등)
  ├─ PII 마스킹 (이메일, 전화번호, 장숫자)
  ├─ URL → 도메인만 보존
  ├─ 민감 키 드롭 (content, body, clipboard 등)
  └─ 길이 제한 (title 64자, URL 128자)
    │
    ▼
SQLite 저장 (store.py)
  배치: 100건 / 1초 flush / 재시도 3회
```

### 2.4 이벤트 데이터 구조
```json
{
  "schema_version": "1.0",
  "event_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "ts": "2026-02-18T09:30:00",
  "source": "os",
  "app": "chrome.exe",
  "event_type": "os.foreground_changed",
  "priority": "P0",
  "resource_type": "window",
  "resource_id": "window_123",
  "payload_json": "{ \"title_hash\": \"abc123\", \"url_domain\": \"google.com\" }",
  "privacy_json": "{ \"masked\": true, \"rules_applied\": [\"url_domain_only\"] }"
}
```

│  대기    │───────────────→│  감시 중   │───────────────→│ 처리 중   │
│ (Idle)  │                │(Watching) │                │(Processing)│

### 2.6 에러 케이스

| 에러 | 원인 | 사용자 메시지 | 복구 |
|------|------|-------------|------|
| 센서 초기화 실패 | Accessibility 권한 부족 | "시스템 접근 권한이 필요합니다. 설정에서 허용해주세요." | 권한 요청 다이얼로그 |
| 큐 오버플로우 | 이벤트 폭주 (>1,000건) | — (내부 처리) | P2 이벤트 자동 드롭 |
| DB 쓰기 실패 | SQLite BUSY | — (내부 처리) | 재시도 3회 (50ms 간격) |
| Denylist 앱 감지 | 비밀번호 관리자 등 | — (무시) | 이벤트 전체 drop |

### 2.7 비기능 요구사항

| 항목 | 목표 | 측정 방법 |
|------|------|-----------|
| CPU 오버헤드 | < 2% (평균) | 수집 중 CPU 사용률 |
| 메모리 사용 | < 30MB | 프로세스 RSS |
| 이벤트 지연 | < 2초 (발생→저장) | 타임스탬프 차이 |
| 이벤트 유실률 | < 0.1% | 입력 대비 저장 비율 |

## 3. F02: Routine — 일일 루틴 분석

### 3.1 기능 개요
수집된 이벤트를 시간 기반으로 세션화하고, 반복 패턴을 루틴 후보로 추출하여 자동화 대상을 식별한다.

### 3.2 3단계 분석 파이프라인

| 단계 | 바이너리 | 입력 | 출력 | 파라미터 |
|------|----------|------|------|----------|
| 1단계 | `build_sessions` | events 테이블 | sessions 테이블 | `--since-hours 6`, `--use-state` |
| 2단계 | `build_routines` | sessions 테이블 | routine_candidates | `--days 3`, `--min-support 2` |
| 3단계 | `build_handoff` | routine_candidates | handoff_queue | `--skip-unchanged`, `--keep-latest-pending` |

### 3.3 루틴 판별 기준
| 기준 | 설명 | 기본값 |
|------|------|--------|
| Support (지지도) | 패턴이 발견된 세션 수 | 최소 2 이상 |
| Confidence (신뢰도) | 패턴의 신뢰도 | 0.0 ~ 1.0 |
| last_seen_ts | 최근 감지 시각 | 90일 이내 |

### 3.4 상태 다이어그램

```
┌──────────┐   3일+ 반복     ┌───────────┐   사용자 승인    ┌───────────┐
│ 이벤트    │──────────────→│ 루틴 후보  │──────────────→│ 활성 루틴  │
│ 수집 중   │               │(candidate)│               │ (active)  │
└──────────┘               └─────┬─────┘               └─────┬─────┘
                                 │                           │
                           90일 미감지                   사용자 비활성화
                                 │                           │
                                 ▼                           ▼
                           ┌───────────┐              ┌───────────┐
                           │  만료     │              │  비활성    │
                           │(expired) │              │(disabled) │
                           └───────────┘              └───────────┘
```

### 3.5 에러 케이스

| 에러 | 원인 | 복구 |
|------|------|------|
| 세션 빌드 실패 | 이벤트 데이터 부족 (<6시간) | 다음 주기까지 대기 |
| 패턴 마이닝 타임아웃 | 대량 세션 데이터 | 배치 크기 축소 후 재시도 |
| 핸드오프 중복 | 동일 패턴 재전송 | `--skip-unchanged` 플래그로 방지 |

## 4. F03: Recommend — 자동화 추천

### 4.1 기능 개요
감지된 루틴 패턴을 기반으로 사용자에게 자동화 스크립트나 워크플로우를 추천한다.

### 4.2 추천 프로세스
| 단계 | 모듈 | 설명 |
|------|------|------|
| 패턴 감지 | `pattern_detector.rs` | 행동 패턴 자동 감지 (23KB+ 로직) |
| Allowlist 추천 | `recommend_allowlist.py` | 수집 허용 앱 목록 자동 제안 |
| 추천 생성 | `recommendation.rs` | LLM 기반 자동화 추천 텍스트 생성 |
| 추천 실행 | `recommendation_executor.rs` | 추천 스크립트 자동 실행 |

### 4.3 추천 출력 예시
```json
{
  "recommendation_id": "rec_001",
  "title": "아침 업무 루틴 자동화",
  "description": "매일 09:00~10:00에 반복되는 메일→브라우저→메모 패턴이 감지되었습니다.",
  "suggested_workflow": {
    "steps": [
      "Gmail 수신함 요약 → Notion 일일 메모에 기록",
      "브라우저에서 주요 뉴스 3건 수집 → 요약"
    ]
  },
  "confidence": 0.82,
  "source_routine": "pattern_morning_routine_001"
}
```

## 5. F04: Control — 앱 내부 제어

### 5.1 기능 개요
특정 애플리케이션 내부를 직접 제어한다 (예: 브라우저 탭 열기, 메일 작성 등).

### 5.2 제어 방식

| 방식 | 모듈 | 우선순위 | 설명 | 대상 OS |
|------|------|----------|------|---------|
| AppleScript | `applescript.rs` | 1순위 | macOS 네이티브 앱 제어 | macOS |
| Accessibility API | `visual_driver.rs` | 2순위 | UI 트리 기반 요소 제어 | macOS/Windows |
| 브라우저 자동화 | `browser_automation.rs` | — | 웹 브라우저 전용 제어 | 크로스 플랫폼 |
| 좌표 기반 | `visual_driver.rs` | 폴백 | 마지막 수단: 화면 좌표 클릭 | 크로스 플랫폼 |

### 5.3 Tool Interface

#### 관찰(Observe) 도구
| 도구 | 입력 | 출력 | 보안 등급 | 설명 |
|------|------|------|-----------|------|
| `ui.snapshot` | scope (string) | JSON | Safe | 활성 윈도우 UI 트리 |
| `ui.find` | query (string) | ElementID[] | Safe | 자연어/속성으로 UI 요소 검색 |
| `screen.capture` | — | Image | Safe | 현재 화면 스크린샷 |

#### 실행(Act) 도구
| 도구 | 입력 | 보안 등급 | 설명 |
|------|------|-----------|------|
| `ui.click` | element_id | Caution | UI 요소 클릭 |
| `mouse.move` | x, y | Caution | 좌표 기반 이동 (폴백) |
| `keyboard.type` | text | Caution | 텍스트 입력 |
| `keyboard.hotkey` | keys[] | Caution | 단축키 실행 |

#### 검증(Verify) 도구
| 도구 | 입력 | 출력 | 보안 등급 | 설명 |
|------|------|------|-----------|------|
| `verify.changed` | before_id, after_id | bool | Safe | 상태 변경 감지 |
| `verify.exists` | element_id | bool | Safe | 요소 존재 확인 |

### 5.4 제어 방식 폴백 체인

```
1순위: AppleScript (macOS 전용)
    │
    ├─ 성공 → 완료
    └─ 실패 ─→ 2순위: Accessibility API
                  │
                  ├─ 성공 → 완료
                  └─ 실패 ─→ 3순위: 브라우저 자동화 (웹 앱인 경우)
                                │
                                ├─ 성공 → 완료
                                └─ 실패 ─→ 최종 폴백: 좌표 기반 클릭
                                              │
                                              ├─ 성공 → 완료
                                              └─ 실패 → 에러 보고
```

## 6. F05: Workflow — n8n 워크플로우 생성

### 6.1 기능 개요
감지된 패턴 또는 사용자 요청을 기반으로 n8n 자동화 워크플로우를 자동 생성·배포한다.

### 6.2 워크플로우 생성 흐름
```
패턴/요청 입력
    │
    ▼
generate_workflow.py / workflow_intake.rs
    │
    ▼
워크플로우 JSON 생성 (workflows/workflow_YYYY-MM-DD.json)
    │
    ▼
n8n API (n8n_api.rs) — 워크플로우 등록/활성화
    │
    ▼
n8n 엔진 (Docker:5678) — 워크플로우 실행
```

### 6.3 상태 다이어그램

```
┌──────────┐   JSON 생성    ┌──────────┐   n8n 등록     ┌──────────┐
│  요청    │──────────────→│  생성됨   │──────────────→│  등록됨   │
│(requested│               │(created) │               │(registered│
└──────────┘               └──────────┘               └─────┬─────┘
                                                            │
                                                      활성화 │
                                                            ▼
                               ┌──────────┐          ┌──────────┐
                               │  실패    │◄─────────│  활성    │
                               │(failed) │  실행 에러 │(active) │
                               └──────────┘          └──────────┘
```

### 6.4 n8n 런타임 옵션
| 모드 | 환경변수 | 사용 환경 |
|------|----------|-----------|
| Docker (기본) | `STEER_N8N_RUNTIME=docker` | 프로덕션 |
| npx | `STEER_N8N_RUNTIME=npx` | 테스트/CI |
| Manual | `STEER_N8N_RUNTIME=manual` | 수동 관리 |

## 7. F06: Exec — 셸 명령 실행

### 7.1 기능 개요
사용자가 요청한 셸 명령을 보안 검사 후 실행한다.

### 7.2 보안 분류
| 등급 | 예시 | 처리 | 사용자 메시지 |
|------|------|------|-------------|
| Safe | `ls`, `pwd`, `echo` | 자동 실행 | — |
| Warning | `cp`, `mv` | Write Lock 해제 필요 | "파일 이동/복사를 실행하려면 승인이 필요합니다." |
| Critical | `rm -rf`, `sudo`, `shutdown` | **차단** (자동 거부) | "위험한 명령이 감지되어 차단되었습니다." |

## 9. F08: NL 자동화 — 자연어 명령 실행

### 9.1 기능 개요
사용자가 자연어로 입력하면, AI가 의도를 파악하고 실행 계획을 세워 OS를 직접 제어한 뒤 결과를 검증하는 end-to-end 파이프라인이다.

### 9.2 실행 파이프라인

| 단계 | 모듈 | 파일 | 역할 | 비기능 목표 |
|------|------|------|------|-------------|
| 1. 의도 분류 | Intent Router | `intent_router.rs` | 자연어 → IntentType | < 1초 |
| 2. 슬롯 추출 | Slot Filler | `slot_filler.rs` | 필수 파라미터 확인/재질문 | < 2초 |
| 3. 계획 생성 | Plan Builder | `plan_builder.rs` | 실행 단계 목록 생성 | < 3초 |
| 4. 실행 | Execution Controller | `execution_controller.rs` | 단계별 OS 액션 실행 | 단계당 < 5초 |
| 5. 검증 | Verification Engine | `verification_engine.rs` | 결과 검증 | < 2초 |
| 6. 승인 | Approval Gate | `approval_gate.rs` | 위험 행동 사용자 승인 | 사용자 응답 대기 |
| 7. 요약 | Summary | `quality_scorer.rs` | 완성도 평가 및 요약 | < 1초 |

### 9.3 Intent 유형
| Intent | 설명 | MVP | 필수 슬롯 |
|--------|------|-----|-----------|
| `FlightSearch` | 항공권/숙박 검색 | ✅ | from, to, date_start |
| `ShoppingCompare` | 가격비교 쇼핑 | ✅ | product_name |
| `FormFill` | 웹폼 자동작성 | ✅ | target_url 또는 현재 페이지 |
| `GenericTask` | 범용 태스크 (fallback) | ✅ | text |

### 9.4 실행 단계 타입
| 타입 | 설명 | 보안 등급 |
|------|------|-----------|
| `navigate` | URL 이동 | Safe |
| `click` | 요소 클릭 | Caution |
| `fill` | 입력 필드 채움 | Caution |

## 10. F09: 항공권/숙박 자동검색

### 10.1 기능 개요
자연어로 항공편 조건을 입력하면 Google Flights에서 자동 검색하고 최적 결과 3개를 요약 제공한다.

### 10.2 슬롯 정의
| 슬롯 | 필수 | 예시 | 타입 |
|------|------|------|------|
| `from` | ✅ | "서울", "ICN" | 도시명/공항코드 |
| `to` | ✅ | "도쿄", "NRT" | 도시명/공항코드 |
| `date_start` | ✅ | "2월 28일" | 날짜 |
| `date_end` | — | "3월 3일" | 날짜 |
| `trip_type` | — | 편도/왕복/다구간 | enum |
| `passengers` | — | 2명 | 정수 |
| `direct_only` | — | 직항만 | boolean |
| `budget_max` | — | 50만원 | 금액 |
| `time_window` | — | 오전/오후 | enum |

### 10.3 실행 흐름
1. 자연어 입력 → 슬롯 추출
2. Google Flights 자동 열기
3. 출발지/도착지/날짜 입력
4. 필터 적용 (예산/시간대/직항)
5. 결과 3개 요약 (가격/경유/시간)
6. 승인 요청 → 상세 페이지 이동

### 10.4 결과 출력 예시

}

- 5가지 시나리오 통과: 직항, 경유, 예산 제한, 시간대 필터, 다구간

## 15. 사용자 시나리오 (Use Cases)

### UC-01: 반복 작업 자동 감지 → 워크플로우 추천
1. 사용자가 매일 아침 메일 확인 → 브라우저 → 메모 앱 순서로 사용
2. Shadow(F01)가 이 패턴을 3일 이상 감지
3. Routine(F02)이 "아침 업무 루틴"으로 세션화
4. Recommend(F03)가 "아침 이메일 요약 워크플로우" 추천
5. 사용자 승인 후 Workflow(F05)가 n8n에 자동 등록

**관련 기능**: F01 → F02 → F03 → F05

### UC-02: 항공권 자연어 검색
1. Chat UI에서 입력: "2월 말에 서울에서 도쿄 왕복 항공권 찾아줘"
2. Intent Router: `FlightSearch` 분류 (confidence: 0.95)
3. Slot Filler: from=ICN, to=NRT, date_start=2/28, trip_type=왕복
4. Plan Builder: Google Flights 검색 계획 생성
5. Execution Controller: 브라우저 열기 → 입력 → 검색
6. 결과 3개 요약: 직항 45만원, 경유 32만원, 아시아나 38만원
7. 승인 → 상세 페이지로 이동

**관련 기능**: F08 → F09 → F04

### UC-03: 셸 명령 실행 (보안 차단)
1. 사용자 입력: "exec rm -rf /tmp/logs"
2. Security Guard: `rm -rf` 감지 → **Critical** 분류
3. 실행 **차단**, 사용자에게 위험 알림 표시

**관련 기능**: F06 (보안 레이어)

### UC-04: 웹폼 자동 작성
1. 사용자 입력: "이 신청서 양식 채워줘"
2. 현재 브라우저 탭의 폼 구조 분석
3. 프로필 매핑: 이름, 연락처, 주소 자동 채움
4. 카드번호 필드 → **차단** (수동 입력 안내)
5. 입력 요약 표시 → 사용자 승인 → 제출

**관련 기능**: F08 → F11 → F04

### UC-05: Telegram 원격 제어
1. Telegram에서 메시지: "오늘 일정 알려줘"
2. Telegram Bot이 메시지 수신 → Core API 전달
3. Google Calendar 조회 → 일정 요약
4. Telegram으로 결과 전송

**관련 기능**: F12 (Telegram + Calendar)

## 19. F16: Subagent/Orchestrator — 멀티 에이전트 오케스트레이션

### 19.1 기능 개요
복잡한 태스크를 여러 서브 에이전트로 분할하고, 오케스트레이터가 전체 실행을 조율한다.

### 19.2 주요 모듈
| 모듈 | 파일 | 역할 |
|------|------|------|
| Orchestrator | `orchestrator.rs` | 태스크 분할 및 서브에이전트 조율 |
| Subagent | `subagent.rs` | 개별 서브 태스크 실행 에이전트 |
| Judgment | `judgment.rs` | 실행 결과 판단 및 의사결정 |

### 19.3 실행 흐름
```
복잡한 태스크 입력
    │
    ▼
Orchestrator (태스크 분할)
    │
    ├──→ Subagent A (검색 태스크)
    ├──→ Subagent B (데이터 추출)
    └──→ Subagent C (결과 종합)
    │
    ▼
Judgment (결과 판단 및 병합)
    │
    ▼
최종 결과 반환
```
