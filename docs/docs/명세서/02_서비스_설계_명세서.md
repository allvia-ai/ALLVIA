### 2.2 에이전트 실행 API

#### POST `/api/agent/execute`
자연어 명령을 실행하고 결과를 반환한다.

| 구분 | 내용 |
|------|------|
| **요청** | `{ text: string, resume_token?: string, resume_from?: string }` |
| **응답** | `{ status: string, logs: [], resume_token: string, completion_score: { score, label, pass, reasons } }` |
| **동시성** | 동일 `plan_id`는 `409 plan_execution_in_progress`로 직렬화 |

**요청 예시**:
```json
{
  "text": "2월 말에 서울에서 도쿄 왕복 항공권 찾아줘",
  "resume_token": null
}
```

**응답 예시**:
```json
{
  "status": "completed",
  "logs": [
    {"step": 1, "action": "navigate", "target": "Google Flights", "result": "ok"},
    {"step": 2, "action": "fill", "target": "출발지", "value": "ICN", "result": "ok"},
    {"step": 3, "action": "extract", "target": "결과", "result": "3건 추출"}
  ],
  "resume_token": "rt_abc123",
  "completion_score": {
    "score": 85,
    "label": "Good",
    "pass": true,
    "reasons": ["검색 완료", "3건 결과 추출 성공"]
  }
}
```

#### GET `/api/agent/preflight`
실행 전 환경을 점검한다 (접근성 권한, 포커스 상태 등).

| 구분 | 내용 |
|------|------|
| **응답** | `{ ready: boolean, issues: [], ax_snapshot: object }` |
| **점검 항목** | Accessibility 권한, 포커스 앱, 브라우저 상태 |

#### POST `/api/agent/preflight/fix`
환경 문제를 자동으로 수복한다.

| 구분 | 내용 |
|------|------|
| **기능** | Finder 전면 복구, 권한 설정 화면 열기, 격리 모드 준비 |

## 3. Collector Service

### 3.1 기본 정보
| 항목 | 값 |
|------|------|
| 구현 | Python (`src/collector/`) + Rust (`collector_rs`) |
| 기본 호스트 | `127.0.0.1` |
| 기본 포트 | `8080` (`STEER_COLLECTOR_PORT`로 변경 가능) |
| DB | SQLite (`steer.db` 공유) |
| 설정 파일 | `configs/config.yaml` |

### 3.2 Collector API

| 엔드포인트 | 메서드 | 설명 |
|------------|--------|------|
| `/events` | POST | 센서 이벤트 수신 및 저장 |
| `/health` | GET | 수집기 헬스체크 |
| `/stats` | GET | 수집 통계 (이벤트 수, 큐 상태 등) |

### 3.3 내장 배치 작업 (Rust Collector)
| 작업 | 주기 | 설명 |
|------|------|------|
| 패턴 워크플로우 생성 | 시작 시 | `workflows/workflow_YYYY-MM-DD.json` |
| 분 단위 집약 | 5분 (`STEER_COLLECTOR_AGG_INTERVAL_SEC`, 기본 300초) | `minute_aggregates` 테이블 갱신 |
| 일일 요약 | 1일 | `daily_summaries` 테이블 갱신 |
| Retention Cleanup | 60분 주기 | 오래된 events, aggregates 삭제 |

### 3.4 수집 설정 (config.yaml 기반)

| 설정 항목 | 값 | 설명 |
|-----------|------|------|
| `queue.max_size` | 1,000 | 이벤트 큐 최대 크기 |
| `queue.shutdown_drain_seconds` | 3 | 종료 시 큐 비우기 대기 시간 |
| `store.insert_batch_size` | 100 | 배치 Insert 단위 |
| `store.insert_flush_ms` | 1,000 | 플러시 주기 |
| `store.busy_timeout_ms` | 5,000 | SQLite 동시 접근 대기 |
| `store.insert_retry_attempts` | 3 | Insert 실패 시 재시도 |
| `store.insert_retry_backoff_ms` | 50 | 재시도 대기 시간 |
| `priority.debounce_seconds` | 2 | 이벤트 디바운스 |
| `observability.log_interval_sec` | 60 | 관측 로그 주기 |

```
요청 수신
    │
    ▼
STEER_API_KEY 설정됨?
    ├─ Yes → 헤더에 Key 포함? ─── Yes → 통과
    │                         └── No  → 401 Unauthorized
    │
    └─ No → STEER_API_ALLOW_NO_KEY=1?
              ├─ Yes → 통과 (로컬 개발 모드)
              └─ No  → 401 Unauthorized
```

## 6. 서비스간 연동 방식

### 6.1 연동 매트릭스

| 연동 경로 | 프로토콜 | 재시도 | 백오프 | 타임아웃 |
|-----------|----------|--------|--------|----------|
| Core → OpenAI | HTTPS REST | 내장 retry | 지수 백오프 | 30초 |
| Core → n8n | HTTP REST | 4회 | 400ms~10s + jitter 10% | 10초 |
| Core → Telegram | HTTPS Bot API | 4회 | 400ms~30s + jitter 10% | 30초 |
| Core → Gmail | OAuth 2.0 + REST | — | — | 15초 |
| Core → Notion | REST API | — | — | 15초 |
| Core ↔ SQLite | 파일 I/O | busy_timeout 5s | — | 5초 |
| Collector → Core DB | SQLite 경로 공유 | 불일치 검증 | — | 5초 |

### 6.2 연동 실패 처리 흐름

```
외부 서비스 호출
    │
    ▼
응답 수신?
    ├─ Yes → 2xx? ── Yes → 성공 처리
    │              └── No → 재시도 가능? (4xx: No / 5xx: Yes)
    │                        ├─ Yes → 재시도 (지수 백오프 + jitter)
    │                        │          └─ 최대 횟수 초과? → 실패 로그 + 폴백
    │                        └─ No  → 즉시 실패 로그 + 에러 반환
    │
    └─ No (타임아웃) → 재시도 (동일 정책)
```

### 6.3 n8n 런타임 설정

|------|----------|------|-----------|

## 9. 장애 대응 시나리오

### 9.1 OpenAI API 장애

```
OpenAI API 응답 없음 / 429 Rate Limit / 5xx 에러
    │
    ▼
LLM Gateway 재시도 (1회, 빠른 폴백 전환 목적)
    │
    ▼
실패?
    ├─ No → 정상 처리
    └─ Yes ─→ 3단계 폴백 체인 실행 (cli_llm::fallback_chat_completion)
              │
              ▼
         ① Gemini CLI 호출
              ├─ 성공 → 결과 반환
              └─ 실패 ─→ ② Codex CLI 호출
                           ├─ 성공 → 결과 반환
                           └─ 실패 ─→ ③ 로컬 llama-server (Ollama) 호출
                                        ├─ 성공 → 결과 반환
                                        └─ 실패 ─→ 사용자에게 알림: "AI 서비스 일시 장애"
                                                     ├─ 이미 생성된 Plan 있음? → Plan 기반 실행 계속
                                                     └─ Plan 없음? → 대기 모드 전환
```

> **폴백 체인 순서**: `STEER_LLM_FALLBACK_CHAIN` 환경변수로 커스터마이징 가능 (기본값: `gemini,codex,local`)

### 9.2 SQLite 락 충돌

```
SQLite BUSY 에러 발생
    │
    ▼
busy_timeout (5,000ms) 대기
    │
    ▼
타임아웃 초과?
    ├─ No → 재시도 (최대 3회, 50ms 간격)
    └─ Yes ─→ 이벤트를 메모리 큐에 유지
              → 다음 flush 주기에 재시도
              → 큐 80% 이상 시 P2 이벤트 드롭
```

### 9.3 n8n 컨테이너 크래시

```
n8n 컨테이너 비정상 종료
    │
    ▼
Docker restart: unless-stopped → 자동 재시작
    │
    ▼
재시작 실패?
    ├─ No → 정상 복구
    └─ Yes ─→ STEER_N8N_RUNTIME=npx 폴백 시도
              └─ 실패 시 → 워크플로우 생성만 로컬 저장
                          → handoff_queue에 pending 유지
                          → n8n 복구 시 자동 전송
```

### 9.4 Core API 크래시

```
Core API 프로세스 비정상 종료
    │
    ▼
Guardian Daemon 감지 (헬스체크 실패)
    │
    ▼
5초 대기 → 프로세스 재시작
    │
    ▼
싱글턴 락 (singleton_lock.rs) 정리 후 정상 기동
    │
    ▼
Desktop App → 자동 재연결 (IPC 복구)
```
