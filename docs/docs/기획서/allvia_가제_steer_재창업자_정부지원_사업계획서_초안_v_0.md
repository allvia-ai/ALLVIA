# 1. Executive Summary (1 Page)

## □ 1-0. 제출용 1페이지 요약(압축본, 숫자/용어 통일본)

### 1) 한 줄 정의
- **ALLVIA는 macOS 로컬 환경에서 OS 실행형 AI 에이전트의 ‘실행 사고’를 Zero-Trust 정책·승인·검증·증빙으로 통제하는 실행 거버넌스(Local Agent Governance) 플랫폼**이다.
- 대상 플랫폼: **macOS 12+ 우선 적용**, Windows 10+는 실행/수집 레이어 추상화 기반으로 단계적 확장(로드맵)

### 2) 문제(Why Customers Pay)
- LLM/Agent 기반 자동화는 확산 중이나, 기업 도입의 핵심 장애는 ‘기능’이 아니라 **오작동/권한사고/감사불가**이다.
- 반복 운영 업무(Ops)는 자동화될수록 ‘누락/부분실패’의 비용과 책임소재(증빙/감사)의 부담이 증가한다.

### 3) 해법(What We Built)
- ALLVIA는 실행을 3단계로 분리하여 “완료 신뢰성(Verified Completion)”을 제품 핵심으로 설계한다.
  1) **Plan(LLM)**: 구조화된 실행 계획(JSON)
  2) **Execute(Rust Core)**: deny-by-default 정책, 위험도 분류(Safe/Caution/Critical), 승인(Write Lock) 통과 시에만 실행
  3) **Verify(Closed-Loop)**: 목표 상태(Assertion) 검증 + Evidence Logging으로 완료 판정

### 4) 차별성(Why We Win)
- **Zero-Trust 5-Layer 실행 게이트**(Chat→Policy→Approval→Outbound→Privacy)
- **Closed-Loop Verification**(‘실행 성공’이 아니라 ‘목표 달성’로 완료 판정)
- **Evidence Logging**(run 단위로 계획/정책판정/승인/검증 근거를 구조화 기록)
- **Shadow Map**(로컬 행동 데이터 요약 기반: 루틴 추천→승인→자동화 승격)

### 5) 초기 타깃(Where We Start)
- 반복 운영 업무가 많고 검증/증빙 니즈가 명확한 팀(Ops/BizOps/Growth Ops/CS Ops)
- 고위험 워크플로우(외부 발신/권한/데이터 변경)를 보유한 조직(승인·감사 가치 즉시 증명)

### 6) PoC(How We Prove) — 합격 기준 기반 실증
- **Design Partner 3~5개 조직, 4~6주 파일럿**
- KPI(핵심): 완료 오판률 **≤ 5%**, 무승인 고위험 실행 **0건**, MTTR **≤ 3분**, 이벤트 유실률 **≤ 0.1%**, 주간 고객당 runs **≥ 30**, 파일럿→유료전환률 **≥ 30%**

### 7) BM & 확장(How We Scale) — 2-Track
- **Track A(OEM 번들링, B2B2C)**: 출하 이미지 프리인스톨(대당 탑재비, 예: ₩5,000/대)로 설치 마찰 0 → 확산 엔진
- **Track B(Team/Enterprise 구독, B2B)**: 좌석 + runs + 거버넌스 패키지(감사/보존/내보내기/정책 프로파일)로 업셀 → 수익 엔진
- OEM 파일럿 KPI(제조사 설득용): Activation **≥ 25%**, D30 Retention **≥ 10%**, 지원 티켓/1,000대 **≤ 20**, 크래시율 **≤ 2**

### 8) Why Now(시장/환경, 통일본)
- RPA 시장(2025): **$4.68B** → 2033 **$35.84B**(CAGR ~29%)
- RPA 온프렘(2025): **$2.09B**(보안/규제 수요)
- Agentic AI는 **2027년까지 40%+ 프로젝트 중단 가능**(가치/리스크 통제 미흡)
- 2025년 글로벌 PC 출하 **270M+ 대**: 엔드포인트 기반 번들링/확산의 기반

### 9) 정부지원 필요성(재창업, 본 과제 구조)
- 지원한도: **정부지원사업비 최대 1억원**
- 협약기간: **7개월 이내**(Phase Gate 기반: MVP→PoC→승격/제품화)
- 정부지원으로 ‘안전성/검증/증빙’ 체계를 선행 구축하여, 상용화의 전제조건(감사/통제)을 확보한다.

# 2. 문제인식(Problem)

## □ 2-1. 시장·산업 환경
1) **Agentic Automation의 확산**: 자연어 → 실행(클릭/입력/브라우저/셸)로 이어지는 흐름이 보편화
2) **기업 도입 장벽의 핵심은 ‘보안/감사/통제’**
   - 자동화 자체는 이미 존재(RPA, 스크립트 등)
   - 그러나 LLM 기반 실행은 “확률적 생성” 특성으로 인해 사고/감사 리스크가 급증
3) **확산 속도는 빠르나 실패도 많다(‘실험→통제’의 전환 구간)**
   - Agentic AI는 2028년까지 기업 소프트웨어에 빠르게 편입될 전망이지만, 동시에 가치/ROI가 불명확한 프로젝트가 대량으로 중단되는 현상도 보고됨
   - 따라서 ‘기능 구현’보다 **검증 가능한 완료(Verified Completion) + 실행 거버넌스(승인/정책/증빙)**가 상용화의 선결조건이 됨
4) **macOS 엔드포인트 확대 및 AI 워크로드 증가(ALLVIA의 로컬 우선 전략의 타당성)**
   - 기업 환경에서 Apple 엔드포인트 비중이 증가하고, AI 워크로드를 macOS 인프라에서 처리하려는 수요가 확대되는 추세
   - 로컬 우선(Local-first) + 프라이버시 가드 + 정책 통제형 에이전트는 이러한 도입 흐름과 정합성이 높음

## □ 2-2. 고객 Pain Point
- 운영 업무는 반복적이지만 ‘검증·증빙’이 동반됨
  - 예: 보고/업데이트/정산/공지/데이터 입력/CRM 업데이트/이슈 트리아지
- 기존 자동화의 한계
  - (1) 실행 실패/부분 실패 시 원인 파악이 어렵고 복구가 느림
  - (2) 외부 발신/권한 액션이 포함되면 승인과 기록이 필요하나 기본 제공이 약함
  - (3) “완료” 판정의 신뢰도가 낮아 결국 사람이 재확인(자동화 ROI 저하)

## □ 2-3. 기존 대안의 구조적 한계
- RPA/매크로: 화면/요소 변경에 취약, 복구 루프/증빙이 약함
- 일반 LLM 에이전트: 실행은 가능하나 **정책 기반 차단/승인/감사/증빙**이 후순위
- 따라서 시장의 공백은 “자동화”가 아니라 **실행 거버넌스(Execution Governance)** 영역

## □ 2-4. 정부지원 필요성(재창업 관점 연결 포인트)
- OS 실행형 에이전트는 안전·검증·감사 체계를 선행 구축해야 상용화 가능(선투자 필요)
- 초기에는 PoC/검증 지표/안전성 기준을 확보하고, 이후 유료 전환(엔터프라이즈/팀 단위)로 확장

# 3. 실현가능성(Solution)

## □ 3-1. 제품/서비스 개요
- ALLVIA는 macOS 로컬에서 동작하는 데스크톱 에이전트로,
  - 사용자는 자연어로 목표(Goal)를 입력
  - 시스템은 계획(Plan) → 정책/승인(Policy/Approval) → 실행(Executor) → 검증(Verify) → 증빙(Logs/Artifacts) 순으로 처리
- 제공 형태(안)
  - (1) 개인/팀용 데스크톱 앱(Tauri)
  - (2) 엔터프라이즈 패키지(정책 프로파일/감사/보안 강화)

## □ 3-2. 핵심 기술 구조(아키텍처)
### 3-2-1. 전체 레이어 개요
- User Input Layer: Tauri Desktop App(React+Vite), CLI, (옵션) Telegram
- API Server Layer: Axum REST API(로컬 127.0.0.1:5680)
- Orchestrator Layer: orchestrator.rs / intent_router.rs / slot_filler.rs
- LLM Gateway Layer: llm_gateway.rs / context_pruning.rs / (옵션) local LLM
- Security Layer(Zero-Trust 5-Layer): chat_gate.rs → policy.rs → approval_gate.rs → outbound_policy.rs → privacy.rs
- Executor Layer: bash_executor.rs / applescript.rs / visual_driver.rs / browser_automation.rs
- Data Collection & Storage: collector_rs, SQLite(steer.db: events_v2, minute_aggregates, daily_summaries, sessions, routines)
- Pattern Analysis & Automation: pattern_detector.rs / recommendation.rs / scheduler.rs
- Workflow Integration(옵션): workflow_schema.rs → n8n_api.rs(워크플로우 CRUD)

### 3-2-2. 안전성 설계의 핵심(Zero-Trust)
- **Deny-by-default**: 변경(write) 계열은 기본적으로 잠김(Write Lock ON)
- **Action Classification**: Safe/Caution/Critical로 분류해 승인 강도 차등
- **Approval Gate**: 위험 액션은 사용자 명시 승인(Human-in-the-loop) 없이는 진행 불가
- **Outbound Policy**: 메일/텔레그램/webhook 등 외부 발신은 별도 정책 검사
- **Privacy Guard**: PII 마스킹(이메일/카드번호 등) 및 민감 앱 denylist 적용
- **Fail-safe**: Kill-switch(긴급 중단), singleton_lock(중복 실행 방지), diagnostic_events(JSONL) 감사 로그

### 3-2-3. Closed-Loop Verification(검증 루프)
- 실행 성공 로그만으로 완료를 인정하지 않음
- OS/앱 상태, 화면 요소, 결과 산출물 등 **Assertion**을 통과해야 완료로 판정
- run 단위로 “왜 완료인지”를 사후 감사 가능한 형태로 저장

### 3-2-4. 서비스 구성(프로세스/포트/라이프사이클)
- ALLVIA는 ‘로컬 머신 내부에서만’ 동작하는 멀티 프로세스 구조로, **외부는 아웃바운드(HTTPS)만 허용**한다.

> (표) 서비스 구성(요약)
>
>| 서비스 | 구현 | 포트(기본) | 역할 | 프로세스/형태 |
>|---|---|---:|---|---|
>| Desktop App | Tauri(React/Vite) | — | 사용자 UI/승인/로그 대시보드 | Steer OS.app |
>| Core API Server | Rust(Axum) | 5680 | 에이전트 실행 엔진 + 로컬 REST API | `local_os_agent` |
>| Collector Service | Python/Rust | 8080 | OS 이벤트 수집 데몬 | `collector_rs` 또는 Python 데몬 |
>| n8n Engine(옵션) | Docker | 5678 | 워크플로우 실행/스케줄링 | `steer-n8n` |
>| Guardian Daemon | Shell | — | 크래시 감지·자동 복구 | `steer-guardian.sh` |
>
- 서비스 생명주기: 시스템 부팅 → Guardian 기동 → Core/Collector/n8n 자동 시작(옵션) → 크래시 시 자동 재시작
- 네트워크 경계: 모든 인바운드 서비스는 `127.0.0.1` 바인딩(외부 접근 차단), 외부 연동은 Outbound Policy Gate를 통과한 경우에만 HTTPS 호출

### 3-2-5. 핵심 API 및 실행 제어(Preflight/Execute/Approval)
- ALLVIA는 “실행 전 점검(Preflight) → 실행(Execute) → 검증(Verify) → 승인(Approval)”을 API 레벨에서 분리하여 운영 가능성을 확보한다.

> (표) 핵심 API(요약)
>
>| 구분 | 엔드포인트 | 기능 |
>|---|---|---|
>| 실행 | `POST /api/agent/execute` | 자연어 명령 실행 및 결과 반환(로그/점수/resume_token) |
>| 사전점검 | `GET /api/agent/preflight` | 접근성 권한/포커스/브라우저 상태 등 환경 점검 |
>| 자동수복 | `POST /api/agent/preflight/fix` | Finder 전면 복구, 권한 설정 화면 오픈 등 수복 |
>| 의도/계획 | `/api/agent/intent`, `/api/agent/plan` | Intent/Slot/Plan 생성(Phase 0) |
>| 검증/승인 | `/api/agent/verify`, `/api/agent/approve` | Assertion 검증 및 승인 게이트 제어 |
>| 품질/릴리즈 | `/api/quality/latest`, `/api/release/gate` | 회귀 검사(릴리즈 게이트) 및 품질 점수 |
>| 운영 | `/api/system/health` | 헬스 체크, DB 경로 불일치 탐지 등 |
>
- 동시 실행 통제: 동일 plan_id 중복 실행은 직렬화 또는 `409 Conflict`로 차단(충돌 방지)
- Resume 모델: 실행 중단/승인 대기 후 `resume_token` 기반으로 동일 지점부터 재개(재현성 확보)

### 3-2-6. 기능 스코프(MVP) 및 의존관계(설계 명세 기반)

### 3-2-7. 로컬 DB 설계 및 보존정책(Privacy by Design)
- DB는 로컬 임베디드 SQLite(WAL) 기반으로 배포 복잡도를 제거하고, 리텐션/마스킹으로 프라이버시-효용 균형을 맞춘다.

> (표) DB 핵심 설계(요약)
>
>| 항목 | 설계 | 목적 |
>|---|---|---|
>| DB 엔진 | SQLite 3 (WAL) | 로컬 앱 배포 최적화/동시성 |
>| Busy Timeout | 5,000ms | 수집기·코어 동시 접근 시 충돌 완화 |
>| 배치 Insert | 100건/1초 flush | 쓰기 성능 최적화 |
>| 크기 상한 | 500MB | 폭증 방지(긴급 정리) |
>| 리텐션 | 원시 7일, 세션 30일, 루틴 90일 등 | 개인정보 최소화 |

> (표) 테이블 생명주기(요약)
>
>| 단계 | 테이블 | 보존 | 용도 |
>|---|---|---:|---|
>| Raw | `events` | 7일 | 원시 이벤트(앱 전환/파일변경/유휴) |
>| Session | `sessions` | 30일 | 시간 기반 세션 요약 |
>| Pattern | `routine_candidates` | 90일 | 반복 패턴 후보(지지도/신뢰도) |
>| Handoff | `handoff_queue` | 7일 | 워크플로우 패키지 큐 |
>| State | `state` | 상시 | 커서/스키마 버전/런타임 메타 |
>| Aggregate | `minute_aggregates`/`daily_summaries` | 설정값/30일 | 대시보드/요약 최적화 |

- 데이터량(예상): 사용자 1명 기준 일 ~2.6MB 증가하나, 리텐션으로 DB는 ~20MB 수준에서 안정화(헤비 유저 시나리오 대비 상한 운영)

### 3-2-8. 운영 안정성(Self-Healing) 및 테스트/릴리즈 게이트
- 안정성은 기능이 아니라 ‘운영 가능한 시스템’의 필수 요건이므로 다음을 기본값으로 내장한다.
  - Guardian Daemon 기반 크래시 감지·자동 재시작
  - singleton lock 기반 중복 실행 방지
  - 재시도/백오프/지터 기반 외부 연동 안정화
  - GUI 회귀 테스트/복합 시나리오 테스트/릴리즈 게이트(회귀 차단)

### 3-2-9. 라이선스/배포 형태(초안)
- 라이선스: MIT(오픈소스 기반 신뢰 확보 + 생태계 확장)
- 배포 형태: (1) 데스크톱 앱(개인/팀), (2) OEM 번들링(B2B2C), (3) 엔터프라이즈 패키지(정책/감사 강화)

## □ 3-3. 데이터 파이프라인(Shadow Map) 및 루틴 승격
- OS 센서(포커스 전환/클릭/타이핑/파일 변화 등) → PrivacyGuard → Normalize/Priority Filter → Session/Routine/Handoff 빌드
- Raw Log(events_v2) → 5분 집계(minute_aggregates) → 일간 요약(daily_summaries)
- Pattern Detector가 반복 패턴을 탐지하여 RoutineCandidate로 승격
- Recommendation Engine이 사용자에게 루틴을 제안하고, 승인 시 워크플로우로 배포

## □ 3-4. n8n 통합(옵션) — 워크플로우 배포/운영
- 내부 루틴을 n8n JSON 스키마로 변환(workflow_schema.rs)
- n8n API Client(n8n_api.rs)로 워크플로우 CRUD 및 실행 트리거(Cron/Webhook)
- 외부 서비스(Gmail/Calendar/Notion/Slack 등) 연동을 노드로 확장

## □ 3-5. PoC 계획 및 검증 설계(초안)
### 3-5-1. PoC 목표(예시)
- 목표1: 고위험 업무(외부 발신/데이터 변경 포함)에서 “승인-검증-증빙”을 기본값으로 제공
- 목표2: 반복 업무에서 루틴 추천 → 워크플로우 승격까지의 전환 퍼널 검증

### 3-5-2. 검증 지표(정량)

- 사용자당 주간 run 수, 루틴 제안 → 승인 전환율

## □ 3-6. TRL 및 단계별 개발 로드맵(초안)
- Phase 1(완료): 기본 실행 루프 + 보안 게이트 + 검증/로그 기반 구축

# 4. 성장전략(Scale-up)

## □ 4-1. 목표시장 정의(Who Pays, Where to Start)
- ALLVIA는 “자동화 도구”가 아니라 **OS 실행형 Agent의 실행 거버넌스(Execution Governance)**를 제공하는 제품으로 정의한다.
- 초기 진입은 ‘도입 장벽이 낮고, 반복 운영 업무가 많으며, 검증/증빙 니즈가 명확한’ 세그먼트를 우선한다.

### 4-1-1. 초기 타깃(1차 ICP)
1) **Ops/BizOps/Growth Ops/CS Ops 팀(10~200명 규모 조직)**
   - 반복 업무(보고/정산/데이터 입력/공지/CRM 업데이트) 비중이 높고, 누락·재작업 비용이 큼
   - 실행 결과를 상시 검증해야 하므로 “완료 신뢰성(Verified Completion)”의 체감 가치가 큼
2) **macOS 기반 업무 비중이 높은 조직(스타트업/디자인·미디어/IT·SaaS)**
   - 엔드포인트가 macOS인 경우, 로컬 에이전트 방식이 배포·운영 측면에서 효율적
3) **고위험 워크플로우를 보유한 팀(외부 발신/권한 변경/데이터 변경)**
   - Approval Gate(Write Lock) + Evidence Logging의 가치를 즉시 증명 가능

### 4-1-2. 확장 타깃(2~3차 ICP)
- **엔터프라이즈 IT/보안/컴플라이언스 조직**: 정책 프로파일/감사/승인 체계가 핵심 구매 포인트
- **규제 산업(금융/헬스케어/법무/공공 등)**: “실행 증빙/감사 가능성”이 도입의 필수 요건
- **MSP/컨설팅/보안 SI 채널**: 고객사 운영 업무 자동화+거버넌스 패키지 형태로 확산

### 4-1-3. 시장 규모 및 성장성(제출용 숫자 세트, 통일본)
- ALLVIA의 목표시장은 **자동화(RPA) × Agentic AI × 엔드포인트 거버넌스**의 교집합이며, 최종 제출본에서는 동일 출처/동일 연도 기준으로 표기한다.

#### (핵심 근거)

- Agentic AI: 2027년까지 **40%+ 프로젝트 중단 가능**(비용/가치 불명확/리스크 통제 부재)
- PC 출하: 2025년 전세계 PC 출하 **270M+ 대** 수준(엔드포인트 기반 소프트웨어 번들링/확산의 기반)

## □ 4-2. 경쟁환경 및 차별화(Competitor & Differentiation)

### 4-2-1. 기존 대안(카테고리)별 한계
1) **RPA/매크로 계열**
   - 화면/요소 변화에 취약하고, 실패 복구 루프 및 증빙이 약함
2) **범용 LLM Agent 계열**
   - 실행은 가능하나, 기업환경에서 요구되는 정책·승인·감사 체계가 후순위
3) **생산성 도구(런처/스크립트/개인 자동화)**
   - 개인 생산성에는 유효하나, 조직 도입에 필요한 통제·감사·표준화 기능이 부족

### 4-2-2. ALLVIA의 구조적 차별성(‘필연성’ 기반)
- **Deny-by-default + Write Lock**: 위험/변경 액션은 기본 잠금, 승인 없이는 실행 불가
- **Action Classification(Safe/Caution/Critical)**: 위험도에 따라 승인 강도/로그 수준을 차등
- **Closed-Loop Verification**: 실행 성공이 아니라 “목표 상태 달성”을 Assertion으로 검증
- **Evidence Logging**: run 단위로 계획-정책-승인-실행-검증 근거를 구조화 기록(감사 가능)
- **Local-first + Privacy Guard**: 로컬 데이터 처리 및 PII 마스킹으로 데이터 반출 최소화

## □ 4-3. 비즈니스 모델(BM) 및 수익화 전략 — 2-Track(번들링 + 구독)

### 4-3-1. BM Track A: PC 제조사 OEM 번들링(B2B2C)
- 제품 탑재를 “설치/온보딩”의 문제로 보지 않고, **출하 단계에서 기본 탑재**하여 CAC와 도입 마찰을 최소화한다.
- 우선 타깃(예시): **주연테크 등 국내 중견 PC 제조사**(출하 이미지 탑재 PoC → 양산 확대)

> (표) OEM 번들링 모델(초안)
>
>| 항목 | 내용 |
>|---|---|
>| 모델 유형 | PC 제조사 OEM 번들링(B2B2C) |
>| 과금 방식 | PC 1대당 탑재 비용(예: ₩5,000/대) |
>| 탑재 형태 | 공장 출하 이미지(프리인스톨) 내장 |
>| 제조사 가치 | ‘AI 내장 PC’ 차별화, 구매 즉시 사용(설치 불필요) |
>| 당사 가치 | 배포 확산(설치 마찰 0), 대량 유입, 고객 데이터/레퍼런스 축적 |
>
- OEM 트랙은 “대중 확산”을 위한 엔진이며, 후속 업셀(Team/Enterprise)로 LTV를 확대한다.

### 4-3-2. BM Track B: Team/Enterprise 구독(B2B)
- 조직 도입은 ‘실행’이 아니라 **정책/승인/감사/보존** 요구가 구매의 결정요인이므로, 엔터프라이즈 패키지로 상향 판매한다.

> (표) 구독 패키지(초안)
>
>| 패키지 | 대상 | 핵심 구성 |
>|---|---|---|
>| Personal | 개인/프리랜서 | 기본 실행 + 기본 로그 + 루틴 추천 |
>| Team | 소규모 팀 | 팀 대시보드, 공유 루틴, 정책 템플릿 |
>| Enterprise | 중대형 조직 | 감사/보존/내보내기, 정책 프로파일, SSO/권한, 배포/운영(MDM 연계) |
>
- 과금 구조(초안): 좌석(Seats) + 실행량(Runs) + 거버넌스 패키지(감사/보존/정책)

### 4-3-3. 업셀/모듈 전략(초안)
- **Compliance Pack**: 로그 보존/내보내기/감사 리포트
- **Connector Pack**: 업무툴 커넥터(노션/슬랙/구글워크스페이스 등)
- **Policy Pack**: 산업/업무별 정책 템플릿(외부발신/권한/결제/파일 변경 통제)

## □ 5-4. 재창업자 고찰: ‘왜 저번엔 망했는가, 이번엔 반드시 성공하는가’

### 5-4-0. 고찰의 전제(심사위원이 요구하는 정답)
- 재창업 심사는 “의지”가 아니라 **(1) 실패 원인에 대한 정확한 규명, (2) 재발 방지 설계의 구체성, (3) 이번 사업에서 성공 조건이 구조적으로 충족되는지**를 본다.
- 따라서 본 항목은 ‘좋은 말’이 아니라, **사업 구조의 실패 가능성을 식별하고, 설계·지표·운영 체계로 제거했음을 증명**하는 데 목적이 있다.

---

### 5-4-1. 왜 저번엔 어려웠는가(실패 원인의 구조적 분해)
이전 사업(하드웨어/인프라 중심)은 ‘기술’이 아니라 **사업 구조**가 실패 확률을 높이는 조건을 내장하고 있었다. 핵심은 아래 3개 병목이 “동시에” 작동했다는 점이다.

#### (1) CAPEX-리드타임-규모의 덫(자본집약 + 장기화)
- 시제품→인증→양산→설치/운영까지의 단계가 길고, 각 단계는 비용과 시간이 선투입된다.
- 일정 지연은 곧바로 **자금 리스크**로 전이되고, 제품 완성도와 무관하게 ‘버틸 수 있는 시간’이 사업 성패를 좌우했다.

#### (2) 외부 의존성이 성장을 잠근다(파트너/현장/규제)
- 하드웨어/인프라 사업은 OEM/설치 현장/규제/인증/공급망 등 외부 변수에 의해 진척이 제한된다.
- 즉, 내부 실행력이 높아도 **외부 변수가 “속도”를 결정**하며, 이 속도는 고객 레퍼런스 확보와 매출 전환을 늦춘다.

#### (3) 가치 증명(ROI)까지의 시간이 길다(Time-to-Value 문제)
- 초기에 고객이 체감하는 가치는 ‘장기 편익’ 형태로 나타나기 쉬워, 의사결정과 계약 전환이 늦다.
- 그 결과, 레퍼런스가 쌓이기 전에 비용이 누적되고, 성장의 임계점(손익분기)까지 도달하기가 어렵다.

#### (4) 확장 비용이 매출과 같이 커진다(운영 복잡도/고정비 증가)
- 생산/물류/설치/AS가 함께 증가하면서, 성장 속도가 곧 고정비 증가로 연결된다.
- 시장 확장 = 운영 복잡도 확장인 구조에서는 “스케일”이 지렛대가 아니라 부담이 된다.

> 결론적으로, 이전 사업은 ‘기술적 실패’가 아니라 **(자본집약 + 외부의존 + 장기 ROI + 확장비용 증가)**라는 구조적 병목이 결합된 사업이었다.

---

### 5-4-2. 이번엔 왜 반드시 성공할 수 있는가(성공을 강제하는 5가지 구조)
ALLVIA는 단순히 아이템이 바뀐 것이 아니라, **성공 조건을 사업 구조에 내장**했다. 이번에는 “잘 되길 바란다”가 아니라, **안 될 이유를 설계로 제거**한다.

#### (1) ‘실행 사고’를 제품 구조로 봉쇄한다: Execution Governance
- ALLVIA는 LLM Agent의 가장 큰 실패 원인(확률적 실행, 누락, 권한사고)을 다음 구조로 통제한다.
  - Deny-by-default 정책
  - Write Lock(승인 게이트)
  - Closed-Loop Verification(완료 검증)
  - Evidence Logging(증빙/감사)
- 즉, 경쟁이 기능의 많고 적음이 아니라 **‘사고를 막는 시스템’**에서 결정되는 영역을 선점한다.

#### (2) 이번엔 “돈이 나오는 문제”를 푼다: 자동화가 아닌 ‘감사 가능한 완료’
- 기업은 “시간 절감”에만 돈을 내지 않는다. **통제/감사/책임소재**가 해결될 때 예산이 열린다.
- ALLVIA는 다음을 정량 KPI로 제시할 수 있어, 구매·전환 논리가 강하다.
  - 완료 오판률(누락/부분실패) 감소
  - 승인/감사 비용 감소(로그/증빙 자동화)
  - 실패 복구시간(MTTR) 단축
- 이는 단순 편의가 아니라 **리스크 비용(재작업/사고/감사 대응)**을 줄이는 지출 항목이다.

#### (3) 소프트웨어는 ‘속도’가 곧 경쟁력이다: 빠른 반복과 낮은 확장비용
- ALLVIA는 Local-first 소프트웨어 제품으로, 배포·개선 리드타임이 짧고 주 단위 학습/개선이 가능하다.
- 성장(사용량/좌석 증가)이 생산/물류/설치 고정비 증가로 직결되지 않는다.
- 또한 로컬 우선(Local-first)은 데이터 반출을 최소화하여 **도입 장벽(보안 우려)**을 구조적으로 낮춘다.

#### (4) 성장 엔진이 내장되어 있다: Shadow Map 기반 루틴→자동화 승격
- 대부분의 자동화 제품은 ‘사용자가 워크플로우를 만들어야’ 확산된다.
- ALLVIA는 사용자 행동을 로컬에서 요약/집계(Shadow Map)하고, 반복 패턴을 루틴 후보로 제안→승인→자동화로 승격한다.
- 즉, **사용량이 쌓일수록 추천 정밀도와 전환(유료/팀 확장)이 동반 성장**하는 구조다.

#### (5) 대표 역량이 이번엔 ‘직접’ 성과로 연결된다: 해외 투자·M&A 기반의 확장 레버리지
- ALLVIA의 확장은 기술만이 아니라 채널/생태계 결합에서 가속된다.
  - MDM/Endpoint(배포) + SSO/권한(Enterprise) + Workflow 생태계(n8n)
- 대표자의 해외 투자·M&A 경험은 제휴·조건 협상·가치평가·리스크 조항 설계에서 실행 속도를 높이며,
  **해외 시장 진입(Design Partner→채널→Enterprise)에서 레버리지로 직접 작동**한다.

### 5-4-3. ‘성공’을 감이 아니라 지표로 판정한다: Phase Gate + Kill Criteria
이번 사업은 낙관으로 확장하지 않는다. **지표가 통과될 때만 다음 단계로 간다.**

> (표) 단계별 성공 판정 지표(초안)
>
>| 단계 | 핵심 질문 | 합격 기준(예시) | 실패 시 조치 |
>|---|---|---|---|
>| Phase A(M1~M3) | 안전하게 실행 가능한가 | 정책 차단 정상률↑, 무승인 고위험 실행 0 | 정책/승인 UX 강화 |
>| Phase B(M4~M6) | 실제 업무에서 신뢰할 수 있는가 | 완료 오판률↓, MTTR↓ | 검증 Assertion 확대 |
>| Phase C(M7~M9) | 사용량이 전환으로 연결되는가 | 주간 runs↑, 루틴 승인률↑ | 추천 정밀도 개선 |
>| Phase D(M10~M12) | 조직 도입 요건을 충족하는가 | 로그 보존/내보내기, 정책 프로파일 | 엔터프라이즈 패키지 강화 |

#### (추가) Kill Criteria(중단/피벗 기준) — 재창업자에게 가장 강한 신호
- 아래 조건 중 하나라도 충족되면 기능 확장이 아니라 구조 수정 또는 타깃 전환을 우선한다.
  1) PoC에서 완료 오판률이 목표치 이하로 수렴하지 않을 경우
  2) 고위험 워크플로우에서 승인 UX로 인해 사용자가 이탈할 경우(가치 대비 마찰 과다)
  3) 파일럿→유료전환이 KPI 기준에 미달할 경우(지불 의사 부재)

### 5-4-4. 리스크를 인정하고, 선제 통제한다(성공 가능성을 ‘강화’하는 방식)
- ALLVIA의 주요 리스크는 ‘인식하지 못한 기술 리스크’가 아니라, **이미 예상 가능한 운영 리스크**다.

  - 경쟁(대형 플랫폼/툴) 진입

  - LLM 품질 변동 및 실행 실패
- 이에 대한 대응은 ‘나중에’가 아니라 **아키텍처와 프로세스에 내장**한다.
  - 실행 레이어 추상화(Executor 모듈화)로 OS/API 변화 대응
  - 정책/승인/아웃바운드 게이트로 사고 범위 축소
  - 외부 점검(모의침투) + Secure SDLC로 신뢰 확보
  - 검증 Assertion 라이브러리 확장으로 완료 신뢰성 수렴

# 6. 추진계획 및 수행일정(연차/월 단위)

## □ 6-1. 추진전략(개발→검증→사업화의 수렴 구조)
- ALLVIA는 OS 실행형 에이전트의 특성상 “기능 구현”보다 **안전성/검증/증빙의 수렴**이 선행되어야 상용화가 가능하다.
- 따라서 본 과제(또는 본 사업)의 추진전략은 다음 3축을 동시에 달성하는 방식으로 설계한다.
  1) **Reliability(완료 신뢰성)**: Closed-Loop Verification으로 ‘완료 오판’을 줄이는 구조 확립
  2) **Governance(통제/감사)**: Zero-Trust 정책·승인·아웃바운드·프라이버시를 기본값으로 내장
  3) **Adoption(도입/확산)**: Shadow Map 기반 루틴→워크플로우 승격으로 사용량과 전환을 만든다

## □ 6-2. 단계별 목표 및 최종 산출물(Deliverables)

- 최종목표2: 반복 운영 업무에서 **루틴 추천→승인→자동화 승격** 전환 퍼널을 검증
- 최종목표3: 엔터프라이즈 도입을 위한 **정책/로그/보존/내보내기(감사)** 기반을 확보

# 8. 인증·표준·보안/컴플라이언스 계획

## □ 8-1. 보안·프라이버시 기본원칙(제품 설계에 내장)
- ALLVIA는 OS 실행형 특성상 보안/감사 요구가 도입의 전제조건이므로, 다음 원칙을 제품 기본값으로 내장한다.
  1) **Deny-by-default**: 변경(write) 계열은 기본 차단(Write Lock) 및 승인 필요
  2) **권한 최소화(Least Privilege)**: 실행에 필요한 권한만 사용
  3) **증적 기반 감사(Auditability)**: run 단위 Evidence Logging 및 내보내기 지원
  4) **개인정보 최소화(Privacy by Design)**: Privacy Guard(PII 마스킹) 및 보존정책
  5) **Fail-safe**: Kill-switch/중복 실행 방지/실패 복구 루프

### 8-1-1. Privacy Guard 상세(명세 기반)
- **앱 필터링**: Allowlist 앱만 수집(27개), Denylist(비밀번호 관리자/민감 앱) 이벤트는 **저장 전 단계에서 Drop**
- **PII 자동 마스킹**: 이메일/전화번호/12자리 이상 숫자(카드번호 등) 레닥션
- **URL 정책**: 전체 URL 저장 금지, **도메인만 보존**
- **민감 키 드롭**: `content`, `body`, `clipboard` 등 민감 페이로드 키 저장 금지
- **길이 제한**: window title 64자, URL/path 128자 등 제한

### 8-1-2. 데이터 보존(Retention) 정책(제출용 요약)
- `events`(원시): 7일
- `sessions`: 30일
- `routine_candidates`: 90일
- `handoff_queue`: 7일
- 집계 테이블(minute/daily): 설정값 또는 30일
- DB 상한: 500MB(초과 시 긴급 정리), WAL 모드 + batch insert(100건/1초), busy_timeout 5초

## □ 8-2. 제품 보안 체계(운영/개발 프로세스)
- Secure SDLC(초안)
  - (1) 위협모델링(Threat Modeling) → (2) 정책/승인 규칙 리뷰 → (3) 보안 테스트(정적/동적) → (4) 릴리즈 승인
- 공급망 보안(초안)
  - SBOM 생성(의존성 목록화), 취약점 스캔, 코드서명/릴리즈 무결성 검증

## □ 8-3. 인증/표준 로드맵(단계별)

> (표) 인증·표준 로드맵(초안)
>
>| 단계 | 목표 | 적용 항목 | 산출물 |
>|---|---|---|---|
>| Phase A~B | 배포 신뢰성 확보 | Apple Developer ID Code Signing, Notarization/Hardened Runtime | 배포 패키지/배포 절차서 |
>| Phase B~C | 외부 점검 기반 신뢰 확보 | 외부 보안 점검/모의침투(필수 모듈 우선) | 점검 리포트/조치 내역 |
>| Phase C~D | 엔터프라이즈 대응 기반 | 로그 보존/내보내기, 정책 프로파일, 접근통제 기본 설계 | 감사 리포트 템플릿 |

## □ 8-4. 컴플라이언스(감사/보존/데이터 관리) 설계
- **로그 보존 정책(초안)**
  - 원시 로그(events): 7일 보존 후 자동 삭제
  - 요약 로그(minute/daily): 30일(또는 고객 정책) 보존
  - run 증적 로그(evidence): 고객 정책에 따라 90~365일 선택형
- **내보내기(Export)**
  - 감사 요청 시 run 단위 근거(계획/정책 판정/승인/검증)를 패키징하여 제출 가능
- **아웃바운드 통제(DLP-lite)**
  - 외부 발신은 Outbound Policy Gate를 통과해야 하며, 금칙어/민감정보 패턴 차단

# 10. 리스크 분석 및 선제 대응 전략

## □ 10-1. 리스크 관리 원칙(재창업자 관점)
- ALLVIA는 OS 실행 권한을 다루므로, 실패를 숨기지 않고 **리스크를 ‘관측 가능한 지표’로 정의**하여 운영한다.
- 리스크 관리는 기능 개발의 후행이 아니라 **아키텍처/프로세스에 내장**하는 것을 원칙으로 한다.

  - (2) 정책·승인·아웃바운드로 사고 범위를 축소
  - (3) 검증 루프(Assertion)로 완료 신뢰성을 수렴
  - (4) 릴리즈 게이트로 모델/코드 회귀를 차단

# 11. 부록

## □ 부록 A. 아키텍처 다이어그램 목록(제출용 설명)
- 본 사업은 ‘기능 나열’이 아니라 **실행 거버넌스(통제·검증·증빙)**를 구현하기 위한 구조적 설계가 핵심이며, 아래 다이어그램은 그 필연성을 보여준다.

1) **UI 아키텍처**
- Tauri 데스크톱 앱(React/Vite/TypeScript) 기반
- 승인(Approval) 팝업, 실행 로그, 운영 대시보드, 루틴/워크플로우 관리 UI

2) **데이터 파이프라인(Shadow Map)**
- 이벤트 수집(클릭/포커스/앱 전환 등) → Privacy Guard 마스킹 → 로컬 SQLite 저장(events_v2)
- 5분 집계(minute) 및 일간 요약(daily) 생성 → 루틴 후보/추천 엔진 입력 데이터로 활용
- 보존정책(예: 원시 7일, 요약 30일 등) 기반의 프라이버시-효용 균형

3) **보안 아키텍처(Zero-Trust)**
- Chat Gate → Policy Engine(deny-by-default) → Approval Gate(Write Lock) → Outbound Policy → Privacy Guard
- 위험도 분류(Safe/Caution/Critical), denylist(파괴 명령), Kill-switch(긴급중단), 감사 로그

4) **전체 시스템 아키텍처(End-to-End)**
- LLM Plan Orchestrator → Rust Core(정책/승인/실행/재시도/로그) → Native Execution Layer(Accessibility/AppleScript/Shell)
- Closed-Loop Verification(증빙 기반 완료 판정)

5) **n8n 연동 다이어그램(옵션)**
- 루틴을 n8n JSON 워크플로우로 변환하고, 스케줄 트리거/외부 커넥터를 통해 확장
- 단, 외부 발신은 Outbound Policy Gate를 통해 통제(거버넌스 우선)

>| 용어 | 정의 | 적용 목적 |
>|---|---|---|
>| Verified Completion | 실행 로그가 아니라 ‘목표 상태 달성’을 검증(Assertion)한 후 완료로 판정 | 자동화의 신뢰성 확보(누락/부분실패 제거) |
>| Execution Governance | 실행을 정책·승인·검증·증빙으로 통제하는 체계 | 기업 도입 장벽(사고/감사) 해결 |
>| Zero-Trust(deny-by-default) | 기본 차단 후 정책으로 허용하는 모델 | 위험 액션 최소화 |
>| Action Classification | 액션을 Safe/Caution/Critical로 분류 | 승인 강도/로그 수준 차등 |
>| Write Lock | 변경/외부발신/권한 등 고위험 액션을 기본 잠금 | 무승인 사고 방지 |
>| Human-in-the-loop(HITL) | 특정 단계에서 사용자 승인/확인을 요구 | 책임/통제/리스크 감소 |
>| Policy Engine | 액션 허용/차단 규칙(denylist/allowlist/컨텍스트 규칙) | 실행 통제의 중심 |
>| Approval Gate | 위험 액션 실행 전 사용자 승인 UX | 고위험 워크플로우 안전화 |
>| Outbound Policy | 메일/메신저/webhook 등 외부 발신 전 정책 검사 | DLP-lite, 정보유출 방지 |
>| Privacy Guard | 수집/로그에서 PII 마스킹 및 민감 앱/필드 보호 | 프라이버시/보안 준수 |
>| Evidence Logging | run 단위로 계획/정책판정/승인/실행/검증 근거를 구조화 저장 | 사후 감사/책임소재 |
>| Closed-Loop Verification | 실행 후 상태 확인→불일치 시 재시도/복구 | 신뢰도 수렴 |
>| Shadow Map | 로컬 행동 로그를 요약·집계하여 ‘루틴 후보’를 생성하는 데이터 지도 | 루틴 추천/승격(확장 엔진) |
>| Routine Candidate | 반복 패턴에서 추출된 자동화 후보 | 사용량→자동화 전환 |
>| Kill-switch | 즉시 실행 중단(긴급 버튼/데드맨 스위치) | 사고 확산 차단 |
>

## □ 부록 C. 심사 대응 Q&A(요약)

### Q1. ALLVIA는 기존 자동화(RPA)와 무엇이 다른가?
- RPA는 ‘정해진 절차 실행’ 중심인 반면, ALLVIA는 **확률적으로 계획되는 Agent 실행을 통제·검증·증빙**하는 실행 거버넌스 제품이다.
- 핵심 차이는 “실행 성공”이 아니라 **검증 가능한 완료(Verified Completion)**를 기본값으로 제공한다는 점이다.

### Q2. 왜 ‘로컬(macOS)’이 필수인가? 클라우드 에이전트로도 되지 않나?
- OS 실행형 업무는 계정/파일/브라우저/앱 컨텍스트가 엔드포인트에 존재한다.
- 로컬 우선(Local-first)은 (1) 데이터 반출 최소화(보안), (2) 네이티브 제어(정확도), (3) 배포/운영 단순화(엔드포인트 앱) 측면에서 유리하다.

### Q3. LLM이 잘못된 실행을 계획하면 어떻게 막나?
- ALLVIA는 LLM을 “집행자”로 두지 않는다.
  - LLM은 Plan(계획)만 생성
  - Rust Core가 Policy/Approval를 통과한 액션만 실행
  - 실행 후 Verify로 목표 달성을 확인
- 즉, 오작동은 시스템 구조로 흡수/차단된다.

### Q4. 개인정보/보안 이슈는 어떻게 해결하나?
- Privacy Guard로 수집/로그에서 PII를 마스킹하고, 민감 앱/필드를 제한한다.
- 외부 발신은 Outbound Policy Gate를 통과해야 하며, 고위험 액션은 Write Lock 승인 없이는 실행 불가다.
- run 단위 Evidence Logging으로 감사 가능성을 확보한다.

### Q5. ‘승인’이 많으면 사용성이 떨어지지 않나?
- 승인은 모든 액션이 아니라 ‘고위험’에 한정한다(Action Classification).
- 저위험 영역은 자동화 효율을 유지하고, 고위험 영역은 승인으로 사고 비용을 제거한다.
- Phase Gate에서 ‘승인 마찰(Approval Friction)’을 KPI로 관리하여 균형을 맞춘다.

### Q6. 경쟁사(대형 플랫폼)가 같은 기능을 하면 어떻게 대응하나?
- 단순 기능은 모방 가능하나, ALLVIA의 핵심은 **정책/검증/증빙이 결합된 실행 거버넌스**이다.
- 또한 Shadow Map 기반의 루틴 승격은 데이터가 쌓일수록 고도화되는 구조로, 사용량이 곧 진입장벽으로 전환된다.

### Q7. 어떻게 돈을 벌고(수익화), 왜 고객이 지불하나?
- 고객은 시간 절감보다 **리스크 비용(재작업/사고/감사 대응)**에 돈을 지불한다.
- BM은 **2-Track**으로 설계한다.
  1) **OEM 번들링(B2B2C)**: 출하 이미지에 프리인스톨(대당 탑재비)로 설치 마찰 0 → 빠른 확산
  2) **Team/Enterprise 구독(B2B)**: 좌석 + runs + 거버넌스 패키지(감사/보존/정책 프로파일)로 상향 판매
- 확장 전략: OEM 확산으로 사용자 기반을 확보하고, 조직 도입(감사/보존/정책)으로 업셀하여 LTV를 확대한다.

### Q7-1. 성장/수익성 지표는 무엇으로 관리하나?(내부 대시보드 기준)
- 제품/사업 성과는 ‘설치 수’가 아니라 **활성/가동/확장 지표**로 관리한다.
  - **MAU 성장률**: 월간 활성 사용자(MAU) 성장률 **45%**(내부 대시보드)
  - **핵심 기능 가동률(Operation Rate)**: **75% 이상** 유지(전환 선행지표)
  - **NRR(순매출유지율)**: Land & Expand 구조로 **125%**(목표치, 상향 판매 중심)
- 비용/마진(COGS) 방어는 ‘리소스 최적화’로 설계한다.
  - Tauri 기반 최적화로 **대기 메모리 점유율 85% 감소(30~45MB)**, 앱 바이너리 **10MB 미만** 목표
  - 모듈화/자동화로 초기 팀 기준 **개발 생산성 65% 향상**(내부 지표)

### Q8. PoC는 무엇을 어떻게 검증하나?
- 업무 3종(저위험/중위험/고위험)을 선정해 ‘승인-검증-증빙’의 가치를 수치로 증명한다.
- 핵심 KPI는 완료 오판률, 무승인 고위험 실행 0, MTTR, 주간 runs, 전환율이다.

### Q9. 왜 지금(Why Now)인가?
- Agentic AI는 확산이 빠르지만 실패도 많아, 기업은 ‘통제/감사/검증’ 없이는 도입하지 않는다.
- 즉, 지금은 기능 경쟁이 아니라 **거버넌스 경쟁**이 시작되는 시점이다.

### Q10. 재창업자로서 이번에는 무엇이 달라졌나?
- 이전 사업에서 학습한 병목(자본집약/외부의존/장기 ROI)을 제거하고,
- 이번에는 제품·시장·운영 구조로 성공 조건을 내장(Phase Gate, Kill Criteria)하여 낙관이 아니라 통제 가능한 성공으로 접근한다.
