## Slide 14. Function Description (0:55)

기능에 대해 간단히 설명드리겠습니다.

우선 **OS Native Control**은 접근성 API를 기반으로 UI 트리를 직접 제어하여 픽셀 좌표에 의존하지 않는 안정적인 자동화를 구현합니다. 여기에 **Rust Performance**를 결합해 네이티브 컴파일 기반의 빠른 응답 속도와 높은 안정성을 확보했습니다. 사용자는 코딩 없이 말로 명령하는 **Natural Language Interface**를 통해 의도를 전달하면, 시스템이 이를 해석해 적절한 기능으로 라우팅합니다.

보안 측면에서는 **Zero-Trust 아키텍처**를 적용해 위험도 분류, Write-Lock, 가드레일, 킬스위치를 통해 오작동이나 폭주를 원천 차단합니다. 또한 **Closed-Loop 구조**를 통해 실행 이후 UI 트리를 다시 스캔하고 교차 검증하며, 모든 과정을 증거 로그로 남깁니다. 이 과정은 Shadow Map, Privacy Guard, 로컬 암호화, 5분 단위 집계 및 보존 정책을 근거로 설계되었습니다.

더 나아가 **Proactive(Shadow Map)** 기능은 사용자의 행동 패턴을 분석해 자동화를 선제적으로 제안합니다. 그리고 **n8n 워크플로우 자동화**를 통해 자연어 명령을 워크플로우 JSON으로 자동 생성·배포하여, 다양한 외부 서비스와 유연하게 연결할 수 있도록 지원합니다.

## Slide **15–16 통합 설명 (1:50)**

“전체 시스템은 왼쪽에서 오른쪽으로 하나의 폐쇄 루프로 설계되어 있는데요.

(슬라이드 오른쪽으로)

먼저 입력은 CLI, Desktop, Telegram 등에서 들어오고, 로컬 API 서버(Axum + Auth)를 통해 코어로 전달됩니다.

코어에서는 Orchestrator가 Intent Router(4-way 분류)와 Slot Filter를 통해 작업을 구조화합니다.
이후 Zero-Trust Security 레이어를 통과해야 합니다. 여기에는 Chat Gate, Policy Engine, Approve Gate, 그리고 Write-Lock이 포함됩니다.
이 단계에서 위험도가 평가되고, 승인되지 않은 쓰기 작업은 차단됩니다.

(슬라이드왼쪽으로) 

승인이 완료되면 Executor 계층에서 Shell Executor, Browser Automation, Visual Driver를 통해 OS Native Control로 실제 실행됩니다.

실행 이후에는 Closed-Loop 검증이 작동합니다.
OS Sensors가 실행 이벤트를 수집하고, Foreground/Idle 상태와 File Watcher 데이터를 모읍니다.
이 데이터는 Privacy Filter를 거쳐 Collector(Python + Rust)로 들어가고,
SQLite(steer.db)에 events → sessions → routines 구조로 저장됩니다.

결과적으로 이 구조는
입력 → 의도 해석 → 제로트러스트 승인 → (슬라이드 왼쪽으로) 네이티브 실행 → 증거 기반 검증 → 패턴 학습 → (슬라이드오른쪽으로) 자동화 추천까지 이어지는 완전한 폐쇄 루프 아키텍처입니다.”

## Slide 17. 기술스택 (0:40초)

“코어는 Rust로 작성했습니다.
이 시스템은 OS 레벨 제어와 실시간 이벤트 수집을 수행하기 때문에
메모리 안전성과 저지연 실행이 필수입니다.

Rust는 GC 없이 네이티브 성능을 제공하고,
단일 바이너리 배포가 가능해 데스크톱 환경에서 안정적으로 동작합니다.

LLM은 폴백 구조입니다.
기본적으로는 1차 모델을 사용하고,
실패하거나 품질 기준을 통과하지 못할 경우 2차 모델로 전환합니다.

즉, 모델 선택 최적화가 아니라
가용성과 안정성을 위한 다중 안전망 구조입니다.”
